# 外部連結檢查爬蟲 - 系統需求規格書 (Requirements)

本文件詳列「外部連結檢查爬蟲」的各項核心需求、系統架構、容錯機制及開發規範。本文件將隨著專案演進持續更新，作為未來擴充功能與架構修改的最高指導原則。

---

## 1. 專案目的
建立一套高可靠性的網站爬蟲系統，用於自動巡覽特定網站，並找出連向外部（非指定網域）的連結，最終解析並紀錄該外部連結對應的 IP 位址。

---

## 2. 系統基本與資安需求 (Basic & Security Requirements)

* **資安防護規範**：
  * **路徑防禦與輸入安全**：系統載入任何設定檔或處理外部路徑輸入時，必須強制進行安全路徑檢驗（如使用絕對真實路徑比對），以防制路徑遍歷（Path Traversal）或利用符號連結（Symlink）繞過檢驗所造成的敏感檔案外洩漏洞。
  * **安全審查與網域白名單**：系統應具備外部連結的安全稽核需求，支援網域白名單檢驗機制，以利發現並篩選出不在信任白名單內的超連結，防範潛在供應鏈資安風險。
  * **傳輸安全與憑證檢驗**：系統需具備識別並標記明文傳輸協定（如 HTTP 而非 HTTPS）之能力，以利傳輸安全性稽核；系統預設應對連線進行嚴格的 SSL/TLS 憑證鏈校驗，同時需保留針對特定信任之自簽憑證網域的配置豁免通道。
  * **機密性與配置憑證保護**：系統日誌與異常輸出中嚴格禁止明文紀錄或印出任何敏感憑證（如代理伺服器帳密）；載入此類機密配置時，應優先支援自環境變數（Environment Variables）讀取。
  
* **CI/CD 自動化與測試規範**：
  * **自動化整合測試**：專案必須隨附一鍵式自動化整合測試套件，用以在 CI/CD 自動化整合管道中進行爬蟲系統的 End-to-End 功能驗證與容錯校驗。
  * **自動化狀態反饋**：測試套件在完成檢驗後，必須能依照校驗結果（如 Queue 數量統計、DNS 異常捕獲、HTTP 狀態碼及去重聚合結果）自動回傳 Exit Code (0 代表成功，非 0 代表失敗)，以利 CI/CD 系統判定測試狀態。
  * **依賴與環境鎖定**：所有的依賴軟體套件必須明確固定版本號，以確保在 CI/CD 容器中執行時的環境重現性與穩定性。

## 3. 爬蟲核心功能需求 (Crawler Core Functional Requirements)

* **指定起點 (Start URL)**：系統需支援設定單一網址作為網頁爬取的起始起點。
* **廣度優先巡覽 (Breadth-First Search)**：系統在探索網站內部網頁時，必須強制採用**廣度優先搜尋 (BFS)** 演算法進行巡覽，以確保巡覽順序的預測性與深度的精確控制：
  * **逐層探索機制**：爬蟲必須以起始網址為第 0 層，逐層向外探索。在完整處理並爬取完當前深度層級的所有待爬網址之前，不得發起下一深度層級的網頁請求。
  * **佇列順序管理**：所有新探索到的內部連結應依序加入待爬網址佇列的末端，並嚴格遵循先進先出 (FIFO) 的順序進行處理。
  * **深度精確限制**：系統必須依據此 BFS 巡覽邏輯，精確計算每個網頁的探索深度。當網址深度達到配置的「最大爬取深度」限制時，系統僅能解析該網頁所包含的外部連結以完成該層的檢查，不得將該網頁內的任何內部網址加入待爬佇列，以確保深度限制與資源控制能精確且有效地執行。
* **網域內遍歷限制**：爬蟲需限制在「指定之特定網域」內進行網頁巡覽，防止無限制地爬取非指定網域的頁面。
* **重新導向 (Redirect) 處理**：當遇到 HTTP 重新導向時，爬蟲需追蹤跟隨至最終目標網址，並以該「最終實際網址」作為基準網址（Base URL）解析 HTML 中的相對路徑，避免拼湊出無效的連結。
* **外部連結與資源資產偵測與紀錄**：
  * **超連結與外連資源掃描**：系統除掃描網頁中的外部超連結（`<a>` 標籤的 `href`）外，為達成防範**失效連結劫持（Broken Link Hijacking）**等供應鏈資安防護目的，亦必須主動掃描網頁中所參照之各式外連靜態與動態資源。
  * **支援掃描之標籤與屬性**：
    * 載入資源類（`src`）：`<script>`、`<iframe>`、`<img>`、`<embed>` 等標籤之外連資源。
    * 樣式表與關聯類（`href`）：`<link>` 標籤之資源。
    * 表單提交類（`action`）：`<form>` 標籤指向之外連目的地，用以防範表單敏感個資外洩至已失效的外連網域。
    * 物件類（`data`）：`<object>` 標籤所嵌入之外部檔案。
  * **欄位紀錄**：詳細記錄其目標網址、來源網址，並解析該目標外連網址主機之 IP 位址，同時進行 HTTP 存活檢查與白名單稽核。
* **內容過濾與資源節約**：
  * **靜態副檔名過濾**：支援自訂的副檔名忽略規則（如 PDF、ZIP 等），避免對非 HTML 網頁資源發送不必要的請求。
  * **規則排除**：支援利用規則運算式（Regex）排除特定的網址路徑，直接忽略符合規則的連結。
  * **媒體內容攔截**：發送請求後需優先檢查回應的內容類型（Content-Type）。若目標為非 HTML 網頁媒體資源，必須立即中斷連線，停止下載檔案內容以節約頻寬。
* **HTTP 狀態與請求延遲機制**：
  * 系統需精準追蹤並記錄每次網址爬取的 HTTP 狀態碼與連線錯誤。
  * **區分跳過情境與延遲等待**：
    * 針對因「副檔名」或「排除路徑」而被跳過的網址（未實際發出 HTTP 請求），系統應標記為跳過且不進行延遲等待，立即處理下一筆以提升效率。
    * 針對因「媒體內容不符」而在讀取標頭後中斷連線的網址（已實際發出 HTTP 請求），系統應標記為跳過並套用延遲等待，避免對目標伺服器造成連線壓力。
* **網址與外連去重機制**：
  * 巡覽佇列需具備防重機制，確保同一個任務內不重複爬取相同的內部網頁。
  * 外部連結紀錄需具備去重機制，針對相同的「來源網址 -> 目標外連網址」不重複寫入。
* **網域 IP 解析優化**：
  * 解析外連 IP 時，系統需擷取目標主機網域進行 DNS 查詢，而非針對完整網址，以減少 DNS 伺服器查詢負擔。
* **瀏覽器偽裝**：
  * 系統需支援自訂 User-Agent 標頭，預設應偽裝為主流網頁瀏覽器，防止被目標網站因預設連線標頭而阻擋。
* **安全傳輸協定稽核 (is_secure)**：
  * **主動識別明文協定**：系統在探索並紀錄外部超連結與靜態資源時，必須主動分析其傳輸協定（Scheme）是否為安全傳輸協定（如 HTTPS）。
  * **持久化與稽核匯出**：非安全傳輸協定（如明文 HTTP）必須被明確標記並儲存於資料庫中，且必須作為結果匯出（CSV/JSON）時的必要檢視欄位，以利資安管理員快速稽核傳輸安全。
* **智慧型錯誤重試與指數退避**：
  * **錯誤分類重試**：遇到永久性錯誤（如 404、403 等）時直接標記為失敗，不進行任何重試；僅對暫時性連線異常或暫時性伺服器錯誤（如 503、連線逾時）進行重試。
  * **指數退避機制**：重試時應引入退避等待機制，隨重試次數增加等待時間，避免連續請求加重目標伺服器負載。
* **外部連結 HTTP 存活檢查與診斷**：
  * 針對探索到的所有外部連結，系統需進行輕量級的 HTTP 存活探測，並記錄其 HTTP 狀態碼與連線錯誤。
  * **雙層探測與繞過機制**：優先發送 HEAD 請求。若目標網址回傳特定的阻擋狀態碼 (如 400, 403, 405) 或是屬於已知設有反爬蟲限制的大型平台 (如社群媒體等)，系統必須自動降級使用帶有微量傳輸範圍限制 (如僅讀取前 1024 位元組的 Range 標頭) 的 GET 串流請求，以模擬正常瀏覽器特徵並繞過反爬蟲 WAF 機制，減少錯誤的損毀連結判定。
  * **任務級存活檢查快取機制**：為了防止對相同的外部網址重複發起大量 HTTP 連線探測 (如在網站中被重複引用數百次)，系統必須實作任務級的健康檢查快取。當同一個外部連結在當次任務中已被探測過，後續引用應直接複用其結果，以節省系統與外部目標網站的頻寬，降低被封鎖風險。
  * **多工並發檢測機制**：系統在探測單一網頁解析出的外部連結清單時，必須強制採用並發檢測技術，以防止同步依序探測所造成的執行緒阻塞與爬行遲滯，提升檢測整體效能。
* **爬行禮貌與頻率控制**：
  * 系統必須實作可調控的網頁請求延遲機制，並支援對不同主機網域進行頻率限制，防止因過度存取對內部或目標網站伺服器造成非預期的負載壓力與服務中斷。
* **資源限制與死循環陷阱防禦 (Resource Limitation & Crawl Trap Defense)**：
  * 系統必須能針對單一任務配置「最大爬取深度」與「最大抓取頁面數量上限」之硬性限制，在超出閾值時必須主動優雅終止任務，防止因網站結構陷阱（Crawl Trap）導致系統資源耗盡。

## 4. 任務管理與容錯機制 (Job Management & Fault Tolerance)

爬蟲系統面臨長時間運作，必須具備狀態驅動的容錯能力與中斷接續能力。

* **狀態驅動設計**：
  * 任務層級狀態需區分掛起、執行中、暫停、完成與異常。
  * 網址佇列層級狀態需區分等待中、已完成、失敗與跳過。
* **中斷與恢復 (Pause & Resume)**：
  * 系統需能應對執行中的非預期中斷，並提供透過任務識別碼恢復未完成佇列繼續執行的機制。
* **設定檔持久化**：
  * 任務建立時，系統需將該次任務合併之最終執行配置快照儲存至資料庫中。未來恢復執行時，強制讀取資料庫內的配置快照，確保參數一致性。
* **執行狀態安全鎖與強制接管**：
  * 為防止多程序同時恢復同一個任務造成數據競爭或資料庫鎖死，系統在啟動時需進行狀態防呆檢查，拒絕執行狀態為「執行中」的任務。
  * 針對意外卡死的任務，系統需提供強制接管的安全解鎖開關。
* **資料庫併發優化**：
  * 針對頻繁的佇列狀態變更與結果寫入，資料庫連線需啟用高併發寫入優化，減少磁碟 I/O 頻率並避免寫入衝突。
* **任務生命週期管理與多使用者隔離**：
  * **多使用者任務隔離**：系統支援在建立或恢復任務時，將任務綁定至特定的使用者識別碼 (`user_id`)，以利多使用者共享系統時的任務歸屬辨識，且支援依使用者識別碼對所有任務進行列表與查詢過濾。
  * **暫停任務**：支援溫和的暫停信號，在完成當下網址爬取後安全暫停。
  * **刪除任務**：支援清理指定任務的所有佇列與結果，防止資料無限制膨脹。
  * **重置任務**：支援將已完成或失敗的任務恢復至初始狀態，以利重新爬取。

## 5. 系統架構與解耦原則 (System Architecture & Decoupling Principles)

系統範圍涵蓋以下模組，且設計需遵循解耦原則：

1. **網站前台**：提供使用者操作介面與檢視結果報告。
2. **網站後台**：提供管理員進行系統維護與狀態管理。
3. **爬蟲程式核心**：專責網頁抓取、HTML 解析與網域 DNS 查詢。
4. **任務管理器**：負責任務排程、佇列狀態調度、資料持久化與異常處理。

> **⚠️ 核心獨立運作保證 (CLI-First)**：
> 爬蟲核心與任務管理模組必須具備高度獨立性。在無前後台網頁系統的環境下，依然能單獨透過命令列介面 (CLI) 啟動、維護與匯出資料。

## 6. 系統配置與開發環境規範 (System Configuration & Development Environment Specs)

* **配置層級、合併邏輯與安全防禦**：
  * **全域與個別配置優先級**：系統提供全域配置定義預設值，並提供個別任務配置自訂參數。個別任務配置原則上優先權高於全域配置，但必須受到全域安全上下限之規範。
  * **設定值聯集合併規則**：為了確保全域安全政策得以落實，對於以下設定項目，系統必須採用「全域與個別配置聯集合併 (Union)」的邏輯，而非直接覆寫：
    * **忽略副檔名 (`ignore_extensions`)** 與 **忽略正規運算式排除路徑 (`ignore_regexes`)**：任務配置與全域配置之清單自動聯集合併。
    * **信任網域白名單 (`approved_domains`)** 與 **自簽憑證豁免網域 (`ssl_exempt_domains`)**：任務配置、全域配置及環境變數之設定值自動聯集合併。
  * **設定參數安全上下限強制限制 (Limit Enforcement)**：全域配置中可定義 `min_timeout`、`max_timeout`、`min_delay`、`max_delay`、`min_retries` 與 `max_retries` 等上下限安全閾值。若個別任務設定檔中的參數超出此安全閾值，系統在啟動任務時必須主動發出警告，並強制將其數值修正至符合全域安全範圍內以執行。
  * **網域特定延遲之匹配原則**：網域特定延遲時間對照表 (`domain_delays`) 支援子網域匹配，且在比對時必須遵循「最長匹配優先原則」（例如，針對 `sub.example.com` 爬取時，若設定中同時存在 `example.com` 與 `sub.example.com` 的延遲設定，應優先套用較長之 `sub.example.com` 的延遲秒數）。
  * **資安路徑防禦**：全域配置與任務配置的載入必須限制在指定的安全目錄內，系統需使用真實絕對路徑檢查，阻絕路徑遍歷（Path Traversal）或符號連結（Symlink）等資安漏洞。
  * **路徑智慧自動補齊**：為了提升使用易用性，當指定的任務設定檔路徑沒有完整前綴且為單純檔名時，系統必須能自動拼接至指定的安全目錄（如 `job/`），同時不破壞底層的真實絕對路徑安全檢驗。
* **日誌管理**：
  * 系統日誌需同時輸出至主控台與日誌檔案，且日誌級別與路徑需支援動態配置。
* **CLI 操作與匯出導出規範**：
  * **參數命名**：輸出報表或結果匯出的相關參數，強制使用完整長參數命名，不提供單一字母縮寫，以防操作混淆。
  * **導出篩選器**：結果匯出必須支援依狀態進行篩選，包含篩選 DNS 失敗連結、HTTP 異常與連線失敗的損毀連結、以及不屬於網域白名單的未審核外連。
  * **去重聚合導出**：支援外連結果去重聚合，合併顯示相同目標外連的引用來源頁面與出現次數。
  * **多格式支援**：支援將結果以 JSON 格式輸出至標準輸出，或直接匯出為 JSON / CSV 檔案。
* **依賴鎖定**：
  * 專案的軟體套件依賴清單必須明確鎖定版本號，確保部署環境的穩定性與可重現性。

## 7. 本機測試網站與測試案例規格 (Test Sandbox Specifications)

* **測試伺服器**：
  * 為在開發與維護期間進行完整的 End-to-End 功能驗證，本專案需配備模擬測試網頁與 Mock 響應之本機測試伺服器。
  * 該伺服器需能模擬各種複雜的 HTTP 回應，以完整涵蓋與驗證爬蟲在重導向基準路徑修正、超時處理、智慧型重試、副檔名與媒體內容過濾、以及 User-Agent 偽裝等核心需求。
* **自動化測試啟動腳本**：
  * 專案需配備一鍵式自動化測試腳本，用以簡化開發測試與支援 CI/CD 程序。
  * 腳本需在背景啟動本機測試伺服器、清除歷史測試資料、執行爬行任務，並在結束後自動執行結果斷言校驗（檢查去重聚合數量、狀態碼捕獲與 DNS 異常紀錄等）。
  * 無論測試成功與否，皆須在結束前確保本機測試伺服器被安全關閉，並依照測試結果返回對應的 Exit Code (0 代表成功，非 0 代表失敗)，以利與自動化集成流程整合。

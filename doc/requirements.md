本文件詳列「網站連結檢查系統」的各項核心需求、系統架構、容錯機制及開發規範。本文件將隨著專案演進持續更新，作為未來擴充功能與架構修改的最高指導原則。

## 1. 專案目的

建立一套高可靠性的網站爬蟲系統，用於自動巡覽特定網站，同時達成以下兩項核心目標：

1. **外部連結偵測與資安稽核**：找出連向外部（非信任網域）的連結，解析並紀錄該外部連結對應的 IP 位址，並進行 HTTP 存活檢查與白名單稽核。
2. **內部連結健康度診斷**：記錄網站內部每個網頁的爬取狀態（HTTP 狀態碼、連線錯誤），自動分類失效樣態（如伺服器異常、資源遺失、連線逾時等），以利維運人員快速定位與修補站內問題。

## 2. 爬蟲核心功能需求 (Crawler Core Functional Requirements)

### 2.1 網頁巡覽與探索策略

* **指定起點 (Start URL)**：系統需支援設定單一網址作為網頁爬取的起始起點。
* **廣度優先巡覽 (Breadth-First Search)**：系統在探索網站內部網頁時，必須強制採用 **BFS** 演算法進行巡覽，以確保巡覽順序的預測性與深度的精確控制：
  * **逐層探索機制**：爬蟲必須以起始網址為第 0 層，逐層向外探索。在完整處理並爬取完當前深度層級的所有待爬網址之前，不得發起下一深度層級的網頁請求。
  * **佇列順序管理**：所有新探索到的內部連結應依序加入待爬網址佇列的末端，並嚴格遵循先進先出 (FIFO) 的順序進行處理。在讀取資料庫佇列的待爬取項目時，必須強制按佇列識別碼 (ID) 進行升冪排序，以從資料庫層面保障 FIFO 的精確性。
  * **深度精確限制**：系統必須依據此 BFS 巡覽邏輯，精確計算每個網頁的探索深度。當網址深度達到配置的「最大爬取深度」限制時，系統僅能解析該網頁所包含的外部連結以完成該層的檢查，不得將該網頁內的任何內部網址加入待爬佇列，以確保深度限制與資源控制能精確且有效地執行。
* **網域內遍歷限制**：爬蟲需限制在「指定之特定網域」內進行網頁巡覽，防止無限制地爬取非指定網域的頁面。
* **重新導向 (Redirect) 處理**：當遇到 HTTP 重新導向時，爬蟲需追蹤跟隨至最終目標網址，並以該「最終實際網址」作為基準網址（Base URL）解析 HTML 中的相對路徑，避免拼湊出無效的連結。
  * **內部 Cookie-gate 穿透與重導向追蹤**：對於內部的同域跳轉，系統亦必須自動收集伺服器的 `Set-Cookie` 並傳遞給下一次跳轉，以利爬蟲順暢通過需要 SSO 或 Cookie-gate 保護的企業內部網站。
  * **越界重導向防護 (Out-of-bounds Redirection Prevention)**：若重導向的目標網址跨出「目標網域」白名單，爬蟲必須立即中斷跟隨並將其視為外部連結紀錄，嚴禁下載及解析外部網站的內容，以節省資源並防範越界爬取。
* **全方位網頁元素解析 (Comprehensive Element Parsing)**：在進行 HTML 解析與連結擷取時，除了常規的 `<a>`, `<img>`, `<script>` 等標籤，爬蟲核心必須嚴格納入現代 Web 多媒體標準，包含擷取 `<video>`, `<audio>`, `<picture>` 內部的 `<source>` 與 `<track>` 標籤之屬性（如 `src`）。這確保系統能完整涵蓋現代網頁的隱含資源，避免遺漏多媒體內容的斷鏈盲區。
* **`<base>` 標籤基準網址解析支援**：在 HTML 解析與連結擷取過程中，若 HTML 內含有 `<base href="...">` 標籤，爬蟲核心解析 HTML 時必須主動讀取該標籤之 `href` 屬性，並將其與當前實際網址（Base URL）合併，做為該網頁所有相對連結路徑解析與正規化之最高優先級基準網址。
* **資源限制與死循環陷阱防禦 (Resource Limitation & Crawl Trap Defense)**：系統必須能針對單一任務配置「最大爬取深度」與「最大抓取頁面數量上限」之硬性限制，在超出閾值時必須主動優雅終止任務，防止因網站結構陷阱（Crawl Trap）導致系統資源耗盡。
* **網址去重機制**：巡覽佇列需具備防重機制，確保同一個任務內不重複爬取相同的內部網頁。

### 2.2 外部連結偵測與紀錄

* **超連結與外連資源掃描**：系統除掃描網頁中的外部超連結（`<a>` 標籤的 `href`）外，為達成防範 **失效連結劫持（Broken Link Hijacking）** 等供應鏈資安防護目的，亦必須主動掃描網頁中所參照之各式外連靜態與動態資源。
* **支援掃描之標籤與屬性**：
  * 載入資源類（`src`）：`<script>`、`<iframe>`、`<img>`、`<embed>` 等標籤之外連資源。
  * 樣式表與關聯類（`href`）：`<link>` 標籤之資源。
  * 表單提交類（`action`）：`<form>` 標籤指向之外連目的地，用以防範表單敏感個資外洩至已失效的外連網域。
  * 物件類（`data`）：`<object>` 標籤所嵌入之外部檔案。
  * 前端自動跳轉（`content`）：`<meta http-equiv="refresh">` 標籤內所宣告的 Client-side Redirect URL。
* **欄位紀錄**：詳細記錄其目標網址、來源網址，並解析該目標外連網址主機之 IP 位址，同時進行 HTTP 存活檢查與白名單稽核。
* **外連去重機制**：外部連結紀錄需具備去重機制，針對相同的「來源網址 -> 目標外連網址」不重複寫入。系統應在 Crawler DB 的 `external_links` 資料表層級對 `(job_id, source_url, target_url)` 加上實體 UniqueConstraint 約束，且在插入資料庫前於程式中預先進行去重查重，防止在高並發或重試機制下寫入衝突。
* **網域 IP 解析與快取優化**：解析外連 IP 時，系統需擷取目標主機網域進行 DNS 查詢，而非針對完整網址，以減少 DNS 伺服器查詢負擔。此外，DNS 查詢結果必須在應用層實作存活時間 (TTL) 快取機制（例如 5 分鐘過期），確保在長時間的爬蟲任務中，若目標網站因 CDN 或負載平衡而切換 IP，爬蟲能及時重新查詢並命中最新 IP，避免大量過期 IP 所引發的連線誤判。
* **安全傳輸協定稽核 (is_secure)**：
  * **主動識別明文協定**：系統在探索並紀錄外部超連結與靜態資源時，必須主動分析其傳輸協定（Scheme）是否為安全傳輸協定（如 HTTPS）。
  * **內部連結明文警示 (Mixed Content Detection)**：除了外部連結外，系統在探索「內部連結」時，若發現屬於同網域但使用了明文的 `http://` 傳輸協定（而非 HTTPS），系統必須對此類連結特別進行標記或記錄警示，協助管理員快速識別未加密的傳輸，防範 Mixed Content 資安風險。
  * **持久化與稽核匯出**：非安全傳輸協定（如明文 HTTP）必須被明確標記並儲存於資料庫中，且必須作為結果匯出（CSV/JSON）時的必要檢視欄位，以利資安管理員快速稽核傳輸安全。
* **目標網域自動信任與邊界判定 (Target Domains Implicit Trust)**：
  * **隱式信任設計**：為了防止目標網域內的內部連結被重複判定為外部連結而送往存活探測引擎，系統在分類連結時，必須將「目標網域 (`target_domains`)」自動且強制視為「信任網域 (`trusted_domains`)」的一部分。
  * **外部連結定義**：只有同時不屬於目標網域且不屬於信任網域的連結，才會被歸類為外部目標連結（External Target Link）並寫入外部連結診斷紀錄中，以消弭邊界網域配置交集造成的雙重判定與重複稽核。

### 2.3 內容過濾與資源節約

* **大檔案攔截與記憶體保護**：爬蟲下載網頁內容時，為防止伺服器回傳無限資料流導致記憶體耗盡 (OOM)，必須強制採用分塊讀取串流 (Chunked Stream) 模式，且須設定「最大下載容量上限」(如 10MB)。一旦資料超過上限，必須立即截斷並只保留已下載的內容。
* **靜態副檔名過濾**：支援自訂的副檔名忽略規則（如 PDF、ZIP 等），避免對非 HTML 網頁資源發送不必要的請求。
* **非必要 `<link>` 資源略過與防呆 (Non-essential Link Exemption)**：在解析 HTML 時，必須主動略過 `<link>` 標籤中屬性包含 `rel="preconnect"`, `rel="dns-prefetch"`, `rel="preload"` 或 `rel="alternate"` 的資源提示或備用文件。排除這些通常僅指向網域根目錄、無效二進位檔或非主要爬取目標的資源，可有效避免爬蟲產生 404 誤判、浪費頻寬下載無意義檔案，以及污染最終診斷報表。
* **規則排除**：支援利用規則運算式（Regex）排除特定的網址路徑，直接忽略符合規則的連結。
* **媒體內容攔截**：發送請求後需優先檢查回應的內容類型（Content-Type）。若目標為非 HTML 網頁媒體資源，必須立即中斷連線，停止下載檔案內容以節約頻寬。
* **區分跳過情境與延遲等待**：
  * 針對因「副檔名」或「排除路徑」而被跳過的網址（未實際發出 HTTP 請求），系統應標記為跳過且不進行延遲等待，立即處理下一筆以提升效率。
  * 針對因「媒體內容不符」而在讀取標頭後中斷連線的網址（已實際發出 HTTP 請求），系統應標記為跳過並套用延遲等待，避免對目標伺服器造成連線壓力。

### 2.4 HTTP 請求策略與反爬蟲對策

* **HTTP 狀態追蹤**：系統需精準追蹤並記錄每次網址爬取的 HTTP 狀態碼與連線錯誤。
* **動態瀏覽器指紋與 HTTP 標頭偽裝 (Dynamic Browser Profiling)**：
  * **高擬真特徵輪替**：系統需支援自訂 User-Agent 標頭。若未自訂，系統預設需實作高擬真的動態特徵產生機制，隨機配對現代瀏覽器的專屬 HTTP 標頭（如 `Sec-Ch-Ua` 等），以最大程度繞過基礎反爬蟲檢測。
  * **HTTP/2 與現代壓縮支援**：爬蟲核心引擎需自動偵測環境並啟用 HTTP/2 (`h2` 套件)，並宣告支援現代壓縮技術 (如 `br`, `zstd`)，以降低被識破的機率。（註：遭遇 WAF 阻擋時的特徵降級與剝離策略，詳見 §2.6）
* **HTTP 代理伺服器支援 (Proxy Support)**：系統需支援配置 HTTP 代理伺服器，以隱藏真實來源 IP 或繞過網路限制。為落實機密保護原則，代理伺服器之設定需支援優先自環境變數（如 `CRAWLER_PROXY_URL`）載入，嚴防包含密碼之機敏憑證被明文寫入設定檔或資料庫的任務快照中。
* **爬行禮貌與頻率控制**：
  * 系統必須實作可調控的網頁請求延遲機制，並支援對不同主機網域進行頻率限制，防止因過度存取對內部或目標網站伺服器造成非預期的負載壓力與服務中斷。
  * **網域專屬延遲 (Domain Delays)**：除全域延遲外，支援以字典合併方式為「特定網域」設定專屬的延遲秒數，並支援子網域最長匹配優先原則，精確控管對外部特定主機的連線壓力。
  * **隨機延遲抖動 (Delay Jitter)**：為防範目標網站基於規律請求時間間隔的行為分析阻擋，系統在請求延遲與錯誤退避等待機制上，必須套用隨機延遲抖動，依據配置的抖動比例（預設 20%）動態隨機增減每次的等待秒數。
* **多層級設定合併與優先覆寫機制**：系統執行配置由「環境變數」、「局部 YAML 設定（任務層級）」與「全域系統 YAML 設定（系統層級）」三層合併而成。其覆寫與聯集機制如下：
  * **數值型與單一值參數**（如 timeout, delay, retries, proxy_url）：覆寫優先順序為環境變數 > 任務局部 YAML 配置 > 全域系統 YAML 配置。
  * **列表與集合型參數**（如 SSL 豁免網域）：合併時採取聯集（Union）方式，且環境變數配置時採用英文逗號分隔的字串格式（例如 `CRAWLER_SSL_EXEMPT_DOMAINS`）。

### 2.5 外部連結存活檢查與診斷

* 針對探索到的所有外部連結，系統需進行輕量級的 HTTP 存活探測，並記錄其 HTTP 狀態碼與連線錯誤。
* **高包容性雙層探測與重導向防誤判機制**：外部連結存活探測必須優先發送 HEAD 請求以節省頻寬。但因 HEAD 請求在許多伺服器上（無論是否涉及 Cookie-gate 防護）的重導向回應並不可靠，若直接信任 HEAD 給出的 Location 可能造成重導向迴圈或誤判。因此，若 HEAD 回傳**任何重導向 (3xx)**，或特定的阻擋狀態碼 (如 400, 403, 405)，或因伺服器轉址設定瑕疵回傳異常狀態 (如 404, 406, 501)，系統必須一律自動降級，改以標準的 GET 串流請求對同一網址重新確認。此機制能有效相容老舊或防護過度之伺服器（如 IIS），大幅降低「假死連結 (False Positives)」誤判率。為避免觸發 WAF 誤判，降級探測時不附加 `Range` 標頭，僅利用連線串流 (`stream`) 於讀取標頭後即主動中斷來節省頻寬。
* **Cookie-gate 防護穿透與跨跳 Cookie 傳遞 (Domain-Bucketed Cookies & Subdomain Inheritance)**：
  * **分桶隔離儲存**：部分網站（如採用 Citrix NetScaler / Cookie-gate 防護的站台）會在重導向時設定驗證 Cookie，並要求後續請求必須攜帶該 Cookie 才能取得正式回應。系統必須在重導向迴圈中自動收集並傳遞跨跳的 `Set-Cookie`，並**依據 Cookie 自行宣告的網域 (c.domain) 分桶儲存**，確保不會跨無關網域洩漏，且滿足同源政策與安全機制。為避免底層網路套件（如 `httpx`）之不穩定行為，系統應強制採用手動組裝 `Cookie` 標頭的方式進行狀態傳遞。
  * **無網域宣告之 Cookie 回退機制 (Domain-less Cookie Fallback)**：針對跨跳轉的 `Set-Cookie` 收集，當伺服器回傳 Cookie 卻未顯式宣告 `Domain` 屬性時，系統絕不可將其丟棄。必須嚴格遵循 RFC 6265 規範，將該 Cookie 視為僅對「發起請求之當前來源主機 (Origin Host)」有效之 Session Cookie，並正確綁定至該主機名稱進行分桶隔離儲存。
  * **子網域萬用字元繼承 (Subdomain Wildcard Match)**：為了防止在同體系不同子網域（如 `acs.elearn.hrd.gov.tw` 與 `elearn.hrd.gov.tw`）間跨跳發生 Cookie 遺失而導致無限重導向，系統在提取分桶 Cookie 時，必須支援**子網域繼承父網域萬用字元 Cookie** 的匹配機制（例如請求 `acs.elearn.hrd.gov.tw` 時，需自動繼承並攜帶來自父網域 `.elearn.hrd.gov.tw` 下的有效 Cookie），以確保 Cookie 穿透的完整性與相容性。
* **社群平台自動降級探測 (Social Domains Fallback)**：專門針對 Facebook、Twitter 等嚴格防護的社群平台網域，當預設的 `HEAD` 請求回傳**任何大於等於 400 的異常狀態碼**時，系統會自動降級為抽離特徵的 `GET` 串流請求進行二次確認，徹底消弭「假死連結」的誤報率。此社群平台網域清單並非寫死，而是由配置參數 `social_domains` 提供，支援管理員依需求進行自訂與擴充。
* **HTTP 明文連線之自動 HTTPS 升級重試機制**：外部連結探測亦適用於突破 HSTS 與 WAF 阻擋之 HTTPS 自動升級重試機制。若升級探測成功，資料庫中依然必須如實記錄來源所寫的原始 `http://` 網址，並將 `is_secure` 標記維持為 `False`，以維持精確的資安稽核功能。（詳細重試邏輯見 §2.6）
* **精細化連線與惡意拖延 (Tarpit) 防禦**：系統需支援分別設定「TCP 連線建立逾時 (`connect_timeout`)」與「外連存活探測總逾時 (`external_check_timeout`)」。確保執行緒能迅速超時釋放，防止爬蟲因等待惡意或極度緩慢的主機而耗盡執行緒池資源。
* **任務級存活檢查快取機制與防 OOM 規範**：為了防止對相同的外部網址重複發起大量 HTTP 連線探測 (如在網站中被重複引用數百次)，系統必須實作任務級的健康檢查快取。**嚴禁在任務啟動或恢復時，將資料庫內的所有探測紀錄一次性預載至記憶體中（防止 OOM 風險）**。必須採用 **延遲載入 (Lazy Loading) 搭配批次查詢 (Batch Query)** 的機制：僅在爬蟲實際發現新的外部連結時，才向資料庫查詢其歷史探測結果。同時，記憶體快取必須採用 **LRU 快取 (如 cachetools.LRUCache)** 並設定明確的安全上限（例如 `maxsize=10000`），以確保無論任務規模多大，記憶體佔用量皆被嚴格限制在安全範圍內，且任務能達成秒速恢復 (Fast Resume)。
* **多工並發檢測機制**：系統在探測單一網頁解析出的外部連結清單時，必須強制採用並發檢測技術，以防止同步依序探測所造成的執行緒阻塞與爬行遲滯，提升檢測整體效能。
* **雙 HttpClient 機制與 SSL 豁免網域子網域繼承**：系統核心內建雙 HttpClient 發送引擎。預設之 Normal Client 套用嚴格的 SSL/TLS 憑證鏈校驗；而針對自簽憑證或開發測試環境，系統支援設定 SSL 豁免網域列表 `ssl_exempt_domains`。當目標網址符合此列表中的網域，系統會自動改用免除憑證驗證之 Exempt Client 發送請求。此豁免機制支援子網域繼承（採用最長匹配優先原則，例如當 `self-signed.org` 被列入豁免，其子網域 `sub.self-signed.org` 將自動繼承豁免行為）。

### 2.6 智慧型錯誤重試與容錯

* **HTTP 明文連線之自動 HTTPS 升級重試機制**：許多現代網站與大型平台防火牆 (如 Cloudflare) 具備 HSTS 規範或嚴格的安全特徵檢查。當爬蟲向明文 `http://` 網址發送內部抓取或外部探測時，常會遭伺服器直接中斷連線或回傳未知的阻擋錯誤。因此，當探測遭遇連線失敗或異常狀態碼 (>= 400) 時，系統必須自動將協定升級為 `https://` 進行二次重試。若 HTTPS 重試遭遇 SSL/TLS 層級錯誤，系統不應退回並掩蓋為原本的 HTTP 錯誤，應如實保留該錯誤並進入後續之 TLS 偽裝降級階段（見下述），利用高擬真 Chrome 指紋嘗試穿透。
* **WAF 防護之降級特徵拔除**：針對任何爬取動作（含 HTTPS），若遭遇特定 WAF 阻擋代碼（如 400, 403, 405, 406 等），系統必須啟動降級重試，主動剝離 `Sec-Ch-Ua`、`Sec-Fetch-*` 等現代瀏覽器特徵標頭再次發送請求，以突破嚴格的企業防火牆限制。
* **雙引擎 TLS 指紋偽裝降級 (Hybrid Strategy)**：當標準的 HTTP 升級與特徵標頭拔除策略均宣告失敗，且伺服器依然回傳高階 WAF 阻擋代碼或直接將連線丟棄（如連線超時/Tarpit，或是 400, 403, 405, 406, 520 等，例如 Cloudflare 的 JS Challenge 攔截）時，系統必須作為最後一道防線，動態載入底層基於 boringSSL 的備援引擎 (`curl_cffi`)，發起 100% 模擬現代瀏覽器 (如 Chrome) 的 TLS 握手與 HTTP/2 框架特徵偽裝請求，以徹底消弭高階機器人防護盾所造成的假死連結誤報。在此偽裝階段中，系統**不會接續中斷的狀態，而是捨棄半路進度並自原始 URL 全盤從頭發起連線**，靠引擎本身嚴格實施跨跳轉的「網域分桶 Cookie 隔離」以防外洩，並確保對於 IPv6 位址能進行正確的格式處理以落實 DNS Pinning。
* **備援連線機制之強制安全繼承 (Fallback Security Inheritance)**：當主連線工具（如 httpx）失敗而觸發底層備援連線機制（如 curl_cffi）進行降級重試時，該備援路徑必須強制且完整地繼承主路徑的所有安全防護與邊界檢查。嚴禁繞過「SSRF IP 阻擋」、「OOM 記憶體下載容量上限機制」及「合法 MIME Type 驗證」，且必須落實底層異常（如 `ValueError`、`OSError`）的捕捉，確保核心無懈可擊。
* **錯誤分類處理與例外統一封裝**：遇到無可救藥的永久性錯誤時直接標記為失敗，避免無效的網路層級重試（僅針對特定連線或防火牆阻擋進行上述特殊重試）。此外，核心引擎的 HTTP 異常與連線例外（如 `HTTPStatusError` 或 `RequestError`）必須完全於底層封裝，統一轉化為結構化的失敗狀態回傳，確保上層調用端免於例外拋出而崩潰。
* **畸形網域與解析容錯 (Malformed Domain Defense)**：當遇到因網頁撰寫瑕疵產生的畸形網址（例如包含連續點號 `..` 導致 DNS 標籤錯誤 `label empty or too long` 之 IDNA 解析例外）時，系統必須精確捕捉其底層例外（如 `UnicodeError`、`ValueError`），並將其優雅地標記為失敗 (`failed`) 且記錄詳細原因，嚴防單一畸形網址穿透防護網導致整個爬蟲任務中斷崩潰。
* **指數退避機制**：重試時應引入退避等待機制，隨重試次數增加等待時間，避免連續請求加重目標伺服器負載。
* **最外層防線與保底機制 (Top-level Exception Boundary)**：所有的背景常駐任務（如排程器迴圈）與爬蟲主程序的最外層，必須實作捕捉所有 `Exception` 的保底機制。若遭遇非預期例外，必須寫入日誌並觸發對應的錯誤狀態轉移（如標記任務為 `error`）或觸發錯誤計數告警，嚴禁任務因未捕捉例外而崩潰進入假死 (Zombie) 狀態或靜默的無窮迴圈。
* **遞迴深度防護 (Recursion Depth Limit)**：任何涉及網路重試或狀態降級（如 HTTP 自動轉 HTTPS 再次探測）的遞迴邏輯，必須具備明確的深度計數器或次數上限，嚴防無限遞迴導致 Stack Overflow。

## 3. 任務管理與容錯機制 (Job Management & Fault Tolerance)

爬蟲系統面臨長時間運作，必須具備狀態驅動的容錯能力與中斷接續能力。

### 3.1 狀態驅動設計

* **子程序管理與狀態持久化**：爬蟲任務的背景執行子程序狀態不應依賴伺服器記憶體變數，應透過寫入實體 PID 檔案機制實作持久化追蹤。系統需能透過 PID 精確判斷子程序是否存活，並於伺服器重啟或異常中斷後能安全地接管或清理。
* **殭屍任務懶加載偵測機制 (Lazy Zombie Job Detection)**：捨棄耗費資料庫 I/O 的傳統定時心跳 (Heartbeat) 機制，改於讀取任務列表時動態比對 `log/pids/` 進程狀態 (`_is_job_running`)。將異常中斷（如 OOM 記憶體不足、伺服器重啟）的任務自動標記為 `error`，優雅且極高效地確保系統狀態一致性。**為避免高頻率 API 呼叫引發不必要的硬碟 I/O 與效能瓶頸，此偵測機制必須實作「時間節流 (Throttling)」保護（例如限制每 30 秒最多執行一次實體掃描）。**
* 任務層級狀態需區分掛起、執行中、暫停、完成與異常。
* 網址佇列層級狀態需區分等待中、已完成、失敗與跳過。

### 3.2 中斷與恢復 (Pause & Resume)

* 系統需能應對執行中的非預期中斷，並提供透過任務識別碼恢復未完成佇列繼續執行的機制。
* **設定檔持久化**：任務建立時，系統需將該次任務合併之最終執行配置快照儲存至資料庫中。未來恢復執行時，強制讀取資料庫內的配置快照，確保參數一致性。
* **主迴圈狀態輪詢節流 (DB Throttling)**：為了避免高頻率查詢癱瘓資料庫 (特別是 PostgreSQL)，爬蟲主迴圈在檢查任務狀態（如偵測協同暫停信號）時，必須實作計數或時間節流機制 (Throttling)，不得在每個網頁迭代時都頻繁發出 SELECT 查詢。

### 3.3 執行狀態安全鎖與強制接管 (Takeover & Force Pause)

* 為防止多程序同時恢復同一個任務造成數據競爭或資料庫鎖死，系統在啟動時需主動檢查，拒絕重複執行狀態為「執行中」的任務。
* 針對疑似卡死（例如爬蟲進程仍在運行，但因不明網路原因陷入停滯或無回應）的任務，系統需提供管理員「強制接管 (Takeover)」機制，將狀態強制重置為 `paused` 以便後續進行重試或重置。
* （註：若任務是因 OOM 記憶體不足或伺服器斷電導致「進程已消失」的假死狀態，則由 §3.1 的「殭屍任務懶加載偵測機制」自動攔截並標記為 `error`，無需人工介入接管。）

### 3.4 任務生命週期管理

* **狀態轉換的原子性保護 (Atomic State Transitions)**：當進行狀態轉換（如啟動任務或檢查並發上限）時，嚴禁使用單純的「先讀取後寫入 (Read-then-Write)」模式。必須強制使用帶有條件的原子性更新 (Atomic UPDATE with Optimistic Locking) 或分散式鎖，防範雙重啟動或突破並發上限的 Race Condition。
* **排隊機制與並發控制 (Queued & Concurrency Control)**：系統實作全域並發任務數限制 (`CRAWLER_MAX_CONCURRENT_JOBS`)。當使用者啟動任務時，若當前系統執行中 (`running` / `starting`) 的任務數量已達上限，新任務將進入 `queued` (排隊中) 狀態。背景排程器會定期輪詢，待資源有餘裕時自動依先進先出原則喚醒任務。
* **暫停與恢復 (Pause/Resume)**：支援溫和的暫停信號，在完成當下網址爬取後安全暫停；並允許後續接續爬取未完成之佇列。針對因 OOM 或伺服器中斷被標記為 `error` 的異常任務，系統亦支援直接無縫恢復執行（CLI 需搭配強制參數），免去重新發起全站爬取的龐大時間成本。**注意：恢復執行時，同樣受系統並發數量上限管控，若資源滿載將先轉為 `queued` 狀態等候調度。**
* **刪除任務 (Delete)**：支援徹底清理指定任務的所有佇列與外連結果，防止資料庫無限制膨脹。
* **重置任務 (Reset)**：支援將已完成或發生中斷的任務恢復至起點，清空佇列狀態與外連紀錄以利重新爬取。
* **失敗重試機制 (Retry Failed)**：支援針對已完成但有部分失敗（包含內部網頁爬取失敗，或外部連結探測異常）的任務進行局部重試，自動將失敗項目重新加入佇列並接續爬取，免去重新爬取整站的龐大開銷，提升維運效率。**同上，重試操作亦受並發數量管控，可能進入 `queued` 狀態排隊。**
* **任務複製 (Duplicate Job)**：支援從歷史任務清單中，一鍵複製既有任務的完整配置參數（包含目標網域、自訂延遲、正則排除規則、社群網域等）並帶入新增任務表單，提升重複排程與定期掃描的維運效率。基於資安防護，複製設定時必須自動遮蔽或清除機敏參數（如 Proxy 密碼），提醒使用者重新輸入。此外，針對在後端採取「聯集與合併」機制的清單與字典型欄位（如忽略副檔名、忽略路徑規則、憑證豁免網域、社群網域與特定網域延遲），在複製任務回填表單時，系統必須與當前全域設定進行比對並濾除相同的部分（維持空值），僅回填使用者真正自訂覆寫的項目，防止全域預設規則被實體硬編碼回填至新表單中。至於其餘單一覆寫型欄位（如總連線逾時、延遲秒數等數值/字串欄位），複製時應直接回填原始快照值，以維持與新建任務表單一致的預設填充體驗。
* **局部重新探測 (Partial Reprobe)**：支援針對非執行中（如 `completed` 或 `error` 等）的任務，將使用者勾選的特定內部頁面或特定外部連結重設為 `pending` 狀態，並清除相關連的舊有結果，隨後將任務自動轉為 `paused` 狀態。管理者可接著點擊啟動任務，讓爬蟲針對此部分進行重新探測。若任務狀態為 `running` 或 `starting`，則系統必須在前端與後端防護並拒絕此操作。
* **CLI 任務生命週期調度指令**：系統核心必須實作完整的任務操作 CLI 介面（含 `--pause`, `--resume`, `--reset`, `--delete`, `--retry-failed`），賦予管理員強大的底層操控力。
* **CLI 結果匯出與多維度篩選器**：系統 CLI 必須實作對任務結果的匯出功能，支援：
  * `--export <job_id>`：指定匯出特定任務之外連結果。
  * `--export-internal <job_id>`：指定匯出特定任務之內部爬行紀錄，用於產出內部頁面失效或診斷報表。
  * `--filter <dead|broken|insecure>`：提供篩選過濾功能。`dead` 篩選 DNS 解析失敗之連結；`broken` 篩選狀態碼 >= 400 或連線錯誤之損毀連結；`insecure` 篩選非安全/明文之連線（`is_secure=False`）。
  * `--exclude <domain>`：在匯出結果中排除特定網域之紀錄。
  * `--group-by <target>`：按目標外連網址進行去重聚合，並統計出現次數（`occurrence_count`）與來源網頁列表。
  * `--json` / `--output <file>`：支援匯出為 JSON 格式（未指定則預設為 CSV 格式）並儲存至指定檔案路徑（未指定則輸出至標準輸出）。
* **CLI 完整報告壓縮包匯出**：CLI 必須實作 `--export-full <job_id>` 命令，能將該任務的「內部爬行紀錄（`crawl_records.csv`）」與「外部連結檢查結果（`external_links.csv`）」完整打包成一個 ZIP 壓縮檔，並儲存於指定輸出路徑，以提供全方位的離線診斷報告。
* **任務資料備份與備援移轉（Backup & Export/Import）**：系統必須提供資料備份與復原移轉之維運工具。支援將特定任務的元資料、爬行佇列與外部連結檢查結果以 JSON 及 JSON Lines 格式（後者可防制大量數據時的記憶體耗盡）匯出，並能自動封裝為標準的 ZIP 壓縮檔。在匯入備份資料時，系統必須能自動指派新的任務 ID，並支援移轉至指定的新使用者所有權下。
* **歷史資料相容與回填機制（Data Schema Compatibility & Backfill）**：為因應資料庫結構演進，系統必須支援資料庫回填工具。當實體 ORM 結構擴充新欄位（例如為 ExternalLink 新增 `target_domain` 欄位）時，系統需提供自動化回填腳本，走訪歷史舊紀錄並自 `target_url` 解析出網域後回填，以確保資料聚合功能在舊任務上亦能正確運行。

### 3.5 多使用者任務隔離與移交

* **多使用者任務隔離**：系統支援在建立或恢復任務時，將任務綁定至特定的使用者識別碼 (`user_id`)，以利多使用者共享系統時的任務歸屬辨識，且支援依使用者識別碼對所有任務進行列表與查詢過濾。
* **任務移交機制 (Job Transfer)**：支援將任務的所有權在不同的管理員/使用者之間移交，移交後原帳號即失去該任務的存取權，以補足多使用者協作下的任務管理閉環。

### 3.6 任務狀態變更 Email 通知

* **觸發時機**：任務狀態轉移至 `completed`（任務正常結束，包含達到最大頁數限制或待爬佇列清空）或 `error`（任務因未預期例外異常終止）時自動觸發。
* **資料庫聯動取得 Email**：在爬蟲進程中，藉由任務的 `user_id`，利用 `backend.auth.db` 在 Auth DB 中查詢取得該使用者的 Email 信箱。若為匿名任務 (`user_id` 為 None)，則不發送通知。
* **統計資訊組裝**：通知郵件（純文字與 HTML 格式）會附帶任務的詳細資訊，並動態統計該任務於 Crawler DB 中的診斷結果，包含：外部連結總數、損壞連結 (Broken Links，HTTP 狀態碼 >= 400) 的數量、失效連結 (Dead Links，DNS 解析失敗) 的數量，以及內部連結失效數量（含爬取失敗與截斷頁面）。
* **模組解耦與防禦性運作 (SRP)**：為符合單一職責原則，報表匯出及發信通知邏輯必須完全自核心管理器 (`manager.py`) 中剝離為獨立的模組 (`exporter.py`, `notifier.py`)。同時，為確保 CLI-First 的獨立運作原則，若爬蟲核心執行於無後端模組的純單機環境中（ImportError），發信模組應能被防禦性跳過，並於日誌中警告，絕不可影響爬蟲子程序的穩定執行。

## 4. 系統架構與解耦原則 (System Architecture & Decoupling Principles)

系統範圍涵蓋以下模組，且設計需遵循解耦原則：

1. **網站前台**：提供使用者登入、任務操作介面與檢視結果報告。
2. **網站後台**：提供管理員進行使用者管理（邀請、停用）、系統設定與任務監控。
3. **爬蟲程式核心**：專責網頁抓取、HTML 解析與網域 DNS 查詢。
4. **任務管理器**：負責任務排程、佇列狀態調度、資料持久化與異常處理。

> **⚠️ 核心獨立運作保證 (CLI-First)**：
> 爬蟲核心與任務管理模組必須具備高度獨立性。在無前後台網頁系統的環境下，依然能單獨透過命令列介面 (CLI) 啟動、維護與匯出資料。CLI 模式下僅需爬蟲資料庫，不依賴帳號資料庫。

### 4.1 資料庫分離架構

系統需維護兩個**實體分離**的資料庫，彼此不得共用連線池或混用 Schema：

| 資料庫 | 用途 | 存取方 |
|--------|------|--------|
| **帳號資料庫**（Auth DB） | 儲存使用者帳號、邀請憑證、Session、角色、登入日誌等身分驗證相關資料 | 前台、後台 Web 層 |
| **爬蟲資料庫**（Crawler DB） | 儲存任務配置、網址佇列、爬取結果、外連紀錄等爬蟲業務資料 | 前台、後台 Web 層、CLI |

* 兩個資料庫的連線設定（路徑 / DSN）需分別獨立配置，透過環境變數或配置檔案各自指定。
* **關聯完整性保護**：針對預設未啟用外鍵檢查的 SQLite，系統建立連線初始化時，必須明確執行 `PRAGMA foreign_keys=ON;`，以確保外鍵約束強制生效。
* **跨庫最終一致性與軟刪除機制**：跨資料庫（Auth DB 與 Crawler DB）資源刪除時，為避免分散式事務失敗導致不一致，系統採「軟刪除 (Soft Delete)」機制，於主庫標記刪除狀態，再由背景任務 (`BackgroundTasks`) 與定期的 Session GC 機制進行實體髒資料清理，確保最終一致性。
* 爬蟲核心與任務管理模組**僅能存取爬蟲資料庫**，不得直接讀寫帳號資料庫；身分驗證邏輯須完全隔離於 Web 層。
* 兩個資料庫的 Schema 遷移腳本需分別管理，可獨立執行。
* **資料庫架構合規檢驗機制（Database Schema Verification）**：為了確保資料庫的真實 Schema 與 Python ORM 模型宣告一致，系統必須提供自動化的架構檢驗工具。該工具能反射（reflect）資料庫實體結構，深入比對並列出缺失的資料表、缺失或型別不符之欄位、缺失索引（含複合索引）及外鍵約束，以防範因部署版本落差引發的執行期例外。
* **跨資料庫引擎資料轉移工具（DB Engine Migration Tool）**：系統必須提供獨立的資料庫遷移工具，支援將本地開發環境使用的 SQLite 帳號與爬蟲任務資料，無縫遷移至生產環境 PostgreSQL 目標資料庫。遷移工具需具備清空重建、資料對映合併與分批遷移以防止記憶體耗盡（OOM）的功能，並在完成數據導入後，自動更新 PostgreSQL 中對應主鍵的自增序列值 (Sequence)，以確保後續資料寫入之主鍵一致性。

### 4.2 跨資料庫 `user_id` 一致性策略

由於 Auth DB 與 Crawler DB 實體分離（§4.1），兩者之間**不得執行跨庫 JOIN**，以下策略確保資料一致性：

* **`user_id` 為 Auth DB 中 `users.id` 的值**，在任務建立時由 Web 服務層從已驗證的 Session 中取得後，寫入 Crawler DB 的 `jobs.user_id` 欄位。Crawler DB 不儲存 any 其他使用者資料（如 email、角色），僅儲存此 ID 以供任務歸屬識別。
* **顯示層組合**：前台 / 後台需呈現「任務所屬使用者 email」時，由 Web 服務層分別查詢兩個資料庫後在應用層組合，禁止在資料庫層跨庫操作。
* **帳號刪除的孤立資料處理 (軟刪除與最終一致性)**：管理員刪除帳號時，Web 服務層會先將 Auth DB 的帳號狀態標記為 `deleted` (軟刪除)，並使其所有 Session 失效。隨後透過背景任務 (`BackgroundTasks`) 或被動觸發的 Session GC 機制，非同步地先清除 Crawler DB 中該 `user_id` 的所有任務及結果資料，最後再實體刪除 Auth DB 中的帳號，以確保最終一致性且不產生孤立資料。跨庫刪除或同步作業若發生例外（如 `SQLAlchemyError`），必須實作明確的告警或日誌通知，嚴防靜默失敗（Silent Failure）。

### 4.3 任務執行觸發方式與程序模型

* **觸發方式**：任務建立後**不自動啟動**；使用者須於前台任務詳情頁面點擊「啟動」按鈕，向後端 API 發送啟動請求。若當下執行中的任務未達上限，即刻啟動；若已達上限，則進入 `queued` 狀態由背景排程器自動調度。
* **執行程序模型**：爬蟲核心以**獨立子程序（Subprocess）** 方式由 Web 服務於收到啟動請求時 fork / spawn，與 Web 服務進程分離執行。
  * Web 服務本身不阻塞等待爬蟲完成，任務狀態由爬蟲子程序直接寫入 Crawler DB。
  * Web 服務透過 Server-Sent Events (SSE) 串流端點，即時讀取 Crawler DB 任務狀態欄位並向前台推送進度更新；當任務進入終態（completed / error）或非執行狀態時，自動關閉串流。此外，後端必須持續監控客戶端連線狀態 (`request.is_disconnected()`)，若偵測到前端斷線需立即釋放串流資源，防範幽靈連線佔用事件迴圈。
  * **SSE 串流防 O(N) 效能瓶頸與 Payload 瘦身**：由於 SSE 需要高頻率輪詢任務進度，為避免觸發高頻的 `O(N)` 資料庫全表掃描 (`COUNT`/`SUM`) 導致效能癱瘓，系統必須實作 In-Memory 記憶體進度快取，並搭配定期節流寫回資料庫（如 `progress_stats` JSON 欄位）的機制，讓 SSE 端點的查詢降為 `O(1)`。同時，SSE 廣播時必須將靜態的大型配置資料（如 `config_json`, `start_url` 等）剔除，僅推送狀態與進度的動態欄位，以大幅節省網路頻寬。
  * 每個任務同一時間只允許一個爬蟲子程序執行（藉由執行狀態安全鎖，詳見 §3.3）。
* **暫停 / 停止信號**：Web 服務透過作業系統信號（如 SIGTERM）或共享狀態欄位向爬蟲子程序傳遞暫停指令，爬蟲在完成當前網址爬取後安全終止。

### 4.4 內部事件驅動解耦架構 (Internal Event-Driven Architecture)

為降低模組間的耦合度並解決循環依賴問題，系統核心導入輕量級的內部事件匯流排 (`events.py`)：
* **業務邏輯與副作用分離 (Separation of Side Effects)**：核心模組（如 Auth 服務或 Crawler 任務管理器）僅專注於執行主要狀態變更，並在完成後發布特定事件（如 `UserDeletedEvent`, `JobStatusChangedEvent`）。
* **通知與日誌解耦 (Decoupled Notification & Audit Logging)**：郵件發送服務 (`notifier.py`) 與稽核日誌服務皆應以訂閱者 (Subscriber) 身分被動接收事件並執行背景操作，嚴禁將發信或日誌寫入邏輯硬編碼 (Hardcode) 於主要業務的函式呼叫鏈中。
* **爬蟲核心依賴反轉 (Dependency Inversion with Callbacks)**：爬蟲核心模組（Crawler Engine）作為獨立引擎，**絕對禁止**主動匯入任何後端（Backend）模組（例如 Web 專用的 `backend.events`）。爬蟲內部所產生的狀態變更，必須透過「依賴注入 (Dependency Injection)」或「Callback 回呼機制」往上層傳遞，以確保爬蟲能在純 CLI 或是單元測試環境中完美獨立運行。
* **避免魔法字串與強型別事件定義 (Type Safety for Events)**：系統內部核心通訊（如 Event Bus 的事件名稱、狀態機轉換等）嚴禁使用散落的純字串（Magic Strings）。必須強制引入 `enum.StrEnum` 或專屬常數類別來統一定義，以確保拼寫檢查安全，並提供 IDE 的自動完成 (Auto-completion) 與重構支援。
* **解決同步阻塞與輪詢風暴**：如 Server-Sent Events (SSE) 等需要高頻率狀態更新的情境，亦透過集中式輪詢器發布事件，再由各 SSE 客戶端非同步訂閱，藉此消除頻繁且大量的資料庫輪詢查詢，大幅提升執行緒池與事件迴圈的穩定度。

## 5. 資安防護規範 (Security Requirements)

### 5.1 輸入驗證與注入防禦

* **路徑防禦與輸入安全**：系統載入任何設定檔或處理外部路徑輸入時，必須強制進行安全路徑檢驗（如使用絕對真實路徑比對），以防制路徑遍歷（Path Traversal）或利用符號連結（Symlink）繞過檢驗所造成的敏感檔案外洩漏洞。
* **跨站腳本攻擊防禦 (XSS Prevention) 與全面禁用 `innerHTML`**：為達到最高的資安與 CSP 標準，前端專案**全面嚴禁使用 `innerHTML` 屬性**（即使是靜態的安全字串也嚴禁使用）。在建構 DOM 結構時，必須強制使用原生 DOM API（如 `document.createElement`、`appendChild`），並只能透過 `textContent` 賦值動態資料，從根本上杜絕惡意腳本注入風險。
* **後端字串插值與 DOM 偽造安全**：後端程式在處理來自不可信來源（如 HTTP `Location` 標頭、外部回應內容）的資料時，嚴禁直接使用 f-string 或字串拼接來構造 HTML 標籤（如 `<a href="{url}">`）。為落實深度防禦 (Defense in Depth)，必須先對不信任的變數進行 HTML Entity 跳脫，或改為直接傳遞資料結構（如陣列、字典）供後續解析，防止潛在的伺服器端 XSS 或解析器注入漏洞。
* **匯出安全與注入防禦 (Export Security & Injection Defense)**：系統在匯出 CSV 報表時，必須強制對所有字串欄位進行跳脫處理 (Sanitization)，防止因惡意網址或內容以 `=`, `+`, `-`, `@` 等字元開頭，導致試算表軟體（如 Excel）將其作為公式執行的 CSV 注入攻擊 (CSV Injection / Formula Injection)。
* **輸入驗證與零信任原則 (Zero Trust)**：落實「不信任前端傳入資料」原則。所有 API 端點的請求參數（包含密碼強度等業務邏輯）需於後端強制進行型別、範圍與規則驗證；接收設定參數時，必須採用 **白名單 (Whitelist)** 機制過濾未知參數，防範繞過前端介面的惡意 API 請求。
* **模糊搜尋與萬用字元跳脫 (LIKE Injection 防禦)**：在使用 ORM 進行字串模糊搜尋（如 `LIKE` 語法）時，除了依賴 Driver 預設的參數綁定防禦 SQL Injection 外，必須強制對使用者輸入中的 `%` 與 `_` 萬用字元進行跳脫 (Escape)，防範攻擊者透過萬用字元構造高成本查詢，導致資料庫全表掃描與效能癱瘓。

### 5.2 跨站請求偽造與安全標頭

* **跨網站請求偽造防禦 (CSRF Protection)**：系統所有狀態變更 API 端點 (POST/PATCH/DELETE) 必須強制實施 Double Submit Cookie 模式的 CSRF 驗證。前端需自 Cookie 讀取 Token 並置入 `X-CSRF-Token` 請求標頭，後端採用常數時間比對校驗，杜絕跨站請求攻擊。
* **內容安全策略與防點擊劫持 (CSP & Security Headers)**：系統 Web 服務必須全域套用防禦性 HTTP 標頭。包含實作嚴格的 `Content-Security-Policy`（搭配每次請求動態生成的 `nonce` 嚴格限制 JavaScript 執行來源）、`X-Frame-Options: DENY`（防範點擊劫持）以及 `X-Content-Type-Options: nosniff`（防 MIME 嗅探）。
* **Cookie 傳輸安全與 CORS 限制**：生產環境下（非除錯模式），系統簽發的 Session 與 CSRF Cookie 必須強制加上 `Secure` 標記（僅限 HTTPS 傳輸）；同時，生產環境必須強制關閉跨來源資源共用 (CORS)，僅允許同源請求，徹底防堵惡意第三方網站的前端呼叫。

### 5.3 SSRF 防禦與傳輸安全

* **SSRF 防禦與內網 IP 阻絕 (DNS Rebinding 防護)**：為了防止伺服器端請求偽造 (SSRF) 與 DNS 重綁定攻擊，系統必須實作嚴格的 IP 驗證。在發送 HTTP 請求前，必須先解析目標網域並驗證該 IP 是否為安全的公網 IP（明確過濾掉 Localhost、Private IP、Loopback 及 Link-Local 位址）。確認安全後，應實作 Socket 層級的 DNS 解析攔截或覆寫 (Monkey Patch) 機制，強制客戶端直接連線至已驗證之 IP（若為 IPv6，必須遵循底層引擎之括號包覆規範），確保絕對安全。在開發測試或受控之本機測試環境下，系統需支援透過設定環境變數 `CRAWLER_ALLOW_LOCAL_IPS=true` 來例外豁免此防禦限制，允許爬蟲探測或爬取 Localhost 與內部私人 IP。
* **傳輸安全與憑證檢驗**：系統需具備識別並標記明文傳輸協定（如 HTTP 而非 HTTPS）之能力，以利傳輸安全性稽核；系統預設應對連線進行嚴格的 SSL/TLS 憑證鏈校驗，同時需保留針對特定信任之自簽憑證網域的配置豁免通道。
* **安全審查與網域白名單**：系統應具備外部連結的安全稽核需求，支援網域白名單檢驗機制，以利發現並篩選出不在信任白名單內的超連結，防範潛在供應鏈資安風險。
* **跨越安全邊界之 Cookie 阻斷與跳轉防護 (Cross-Boundary Redirect Mitigation)**：為防範中間人攻擊 (MITM) 與 Cookie 注入 (Cookie Injection) 風險，當爬蟲從「SSL/TLS 憑證豁免網域 (ssl_exempt_domains)」重導向跳轉至「非豁免網域（需嚴格驗證）」時，系統必須實作嚴格的安全邊界檢查。一旦偵測到此類跨越安全邊界的跳轉，除了下一跳必須強制恢復憑證驗證外，系統更應主動阻斷並丟棄該次不安全連線所獲取的 Cookie 狀態，嚴禁其繼承至下一跳，以徹底防禦 Session Fixation 攻擊。

### 5.4 機密保護與資訊洩漏防禦

* **機密性與配置憑證保護**：系統日誌與異常輸出中嚴格禁止明文紀錄或印出任何敏感憑證（如代理伺服器帳密）；載入此類機密配置時，應優先支援自環境變數（Environment Variables）讀取。
* **最小權限暴露 (Masking)**：後端在回傳系統設定或任務快照至前端時，必須主動過濾內部邏輯設定，並將敏感憑證（如 Proxy 密碼）進行遮蔽（Masking）處理。
* **統一例外攔截與堆疊隱藏 (Stack Trace Hiding)**：Web 服務層必須實作全域例外攔截器 (Global Exception Handler)。當捕捉到未處理的伺服器內部錯誤時，一律回傳標準的 HTTP 500 JSON 錯誤訊息，嚴禁將系統內部的堆疊追蹤 (Stack Trace) 或套件版本暴露於前端，防止架構資訊外洩。
* **防禦計時攻擊 (Timing Attack)**：系統在處理任何敏感憑證的驗證（如登入密碼驗證、CSRF Token 比對）時，必須確保執行時間具備恆定性。字串比對須強制使用安全的時間恆定比較函式（如 `secrets.compare_digest`）；密碼驗證時，無論帳號是否存在，皆須執行等效耗時的雜湊運算，防範攻擊者透過測量回應時間探測帳號狀態。

### 5.5 資源耗盡防禦

* **防呆與系統資源保護 (Resource Exhaustion Defense)**：爬蟲全域配置必須實作安全上下限 (Min/Max Thresholds) 驗證。當個別任務指定的請求延遲 (Delay) 過短，或重試次數 (Retries) 與逾時時間 (Timeout) 過長時，系統 API 層必須能自動攔截並強制將其收斂至全域安全範圍內，防止人為設定錯誤導致自身伺服器資源耗盡或對目標網站造成 DDoS。
* **並發與資料庫連線池防護 (Concurrency & Connection Pool Protection)**：後端 API 由於遵守了將包含資料庫操作的端點宣告為 `def` (同步)，FastAPI 會將其分派至內部 Thread Pool 執行（預設最大 40 個 Thread）。為防止瞬間高併發請求導致 Thread Pool 內的執行緒卡在等待 DB Connection 而引發 503 超時錯誤，系統必須確保資料庫連線池配置（即 `DB_POOL_SIZE` 加上 `DB_MAX_OVERFLOW` 的總和）必定大於或等於 FastAPI 的預設 Thread Pool 數量（如總和 60），徹底消除高併發下的等待瓶頸。

## 6. 使用者帳號管理與邀請制登入機制 (User Account & Invitation-Based Authentication)

本系統採邀請制（Closed Registration），禁止公開自由註冊，所有使用者帳號須由後台管理員建立並發送邀請。

### 6.1 邀請流程

* **後台建立帳號**：管理員於後台輸入受邀者的電子郵件信箱，系統自動產生一組唯一的 UUID 作為一次性邀請憑證，並將帳號狀態設定為「待首次登入（Pending）」。
* **邀請信件寄送**：系統自動組成邀請郵件，內含：
  * 受邀者的使用者帳號（電子郵件信箱）
  * 一次性邀請 UUID（用於身分驗證）
  * 系統登入網址（不包含 Token，防止瀏覽器歷史紀錄或 Referer 洩漏）
  * 邀請碼有效期限（可配置，如 72 小時）
* **邀請憑證安全性**：
  * UUID 須為單次使用；首次登入完成並設定密碼後，邀請 UUID 立即失效，不得再次使用。
  * 超過有效期限的邀請連結應自動失效，管理員可重新寄出新的邀請。

### 6.2 首次登入流程

1. **輸入邀請資訊**：使用者進入系統登入網址後，需手動輸入電子郵件與專屬邀請碼 (UUID) 進行驗證。
2. **身分驗證**：後端驗證電子郵件與 UUID 組合的有效性（帳號存在、狀態為 Pending、UUID 未失效）。
3. **強制設定密碼**：身分驗證通過後，系統立即導向「密碼設定頁面」，使用者**不得跳過**此步驟，在設定符合安全標準的密碼之前，無法進入任何功能頁面。對於透過 CLI 建立且使用初始隨機密碼登入的首位管理員，亦強制適用此攔截與設密機制。
4. **密碼安全標準**：
   * 長度：最少 12 個字元。
   * 複雜度：需同時包含大寫字母、小寫字母、數字、特殊符號中的至少三項。
   * 禁用規則：禁止使用與帳號（電子郵件本地端）相同或過於相似的字串作為密碼。
   * 儲存：密碼需以強雜湊演算法加密後儲存（系統指定採用 `bcrypt`，並需設定具備足夠抵抗暴力破解強度的 work factor，至少為 12），嚴禁明文或弱雜湊（如 MD5、SHA-1）。
5. **完成啟用**：密碼設定完成後，帳號狀態更新為「正常（Active）」，邀請 UUID 立即清除失效，系統自動導向主功能頁面。

### 6.3 日常登入流程

* **一般登入**：已啟用帳號使用電子郵件 + 密碼登入。
* **Session 安全**：
  * 登入成功後簽發具有效期限的 Session Token（或 JWT）。
  * 前端未經身分驗證的請求一律重新導向至登入頁面。
* **登入失敗保護**：連續登入失敗達閾值（如 3 次）後，帳號暫時鎖定（如 15 分鐘），防範暴力破解。
* **Session 背景清理 (Garbage Collection)**：系統於每次觸發登入或登出操作時，應透過背景任務 (Background Tasks) 自動清理資料庫中已過期或失效的 Session 紀錄，防止無效資料庫容量無限制堆積。
* **登出**：提供明確的登出機制，清除 Session / Token。

### 6.4 密碼變更

* 已登入的使用者可主動在「帳號設定」中修改密碼，需先輸入現有密碼進行驗證後才可設定新密碼，新密碼同樣需符合 §6.2 的密碼安全標準。

### 6.5 忘記密碼與重設機制 (Forgot Password)

* **防禦帳號列舉 (Anti-Enumeration)**：使用者輸入 Email 申請重設密碼時，無論該信箱是否存在於系統中，系統皆回傳相同的成功提示與一致的回應時間，徹底防範攻擊者刺探帳號是否存在。
* **申請限速防護 (Rate Limiting)**：為防範惡意濫發重設郵件，系統需針對單一客戶端 IP 實施申請頻率限制（例如 15 分鐘內最多允許 3 次申請，參數可由環境變數動態配置）。
* **Token 安全性與時效**：
  * 產生高強度隨機字串作為重設憑證，為防範資料庫外洩遭直接盜用，資料庫中僅能儲存該 Token 的雜湊值 (Hash)。
  * 重設連結具備嚴格時效性（預設 1 小時），逾期後自動失效。
  * Token 為單次使用，密碼重設成功後該憑證立即作廢。
* **密碼重設與強制登出 (Session Invalidation)**：
  * 使用者點擊連結後，可設定新密碼（新密碼需符合 §6.2 之密碼安全標準）。
  * 密碼重設成功後，為確保帳號安全並防範帳號遭惡意接管，系統必須**自動強制登出該使用者的所有現行有效 Session**，要求其使用新密碼重新登入。

### 6.6 帳號狀態管理

| 狀態 | 說明 |
|------|------|
| Pending | 已建立帳號、邀請信已寄出，尚未完成首次登入設密 |
| Active | 帳號正常可用 |
| Suspended | 由管理員停用，禁止登入 |
| Expired | 邀請連結過期且尚未首次登入，需重新寄送邀請 |

### 6.6.1 使用者角色權限提降管理 (Promote & Demote)

* **角色提降權限**：系統管理員有權限在後台使用者列表中修改其他帳號的角色，將一般使用者提升為管理員（Promote），或將現有管理員降權為一般使用者（Demote，除自己以外）。
* **生效時機**：權限變更會即時寫入 Auth DB，並於該帳號的下一次 API 請求或 Session 續期時即時生效，確保角色存取控制的嚴密性。

### 6.7 帳號資料庫（Auth DB）Schema 規範

帳號相關資料**獨立存放於專屬的帳號資料庫**，與爬蟲資料庫實體分離（詳見 §4.1）。Auth DB 需涵蓋以下資料表：

* **`users`**：儲存使用者基本資料（id、email、password_hash、role、status、created_at、last_login_at）。
* **`invitations`**：儲存邀請憑證（id、user_id、uuid、expires_at、used_at、created_at）；UUID 使用後立即標記 `used_at`。
* **`sessions`**：儲存有效的 Session Token（token_hash、user_id、expires_at、created_at、ip_address、user_agent）；登出後立即刪除或標記失效。
* **`password_reset_tokens`**：儲存密碼重設憑證（id、user_id、token_hash、expires_at、used_at、created_at）；使用後立即標記 `used_at`。
* **`auth_logs`**：儲存登入 / 登出 / 失敗 / 鎖定等身分驗證事件，以及後台敏感的高風險管理操作事件（如全域配置變更、任務強制操作、使用者帳號狀態變更與刪除等）（user_id、event_type、ip_address、detail、created_at）。其 `detail` 欄位以 JSON 格式儲存詳細的操作前後差異與目標資訊。後端敏感操作日誌與查詢 API 必須支援日期範圍篩選參數（如 `from_date`），以利管理員限縮並快速查閱特定區間內的維運事件。

> Session Token 本身不得以明文儲存於資料庫，需儲存其雜湊值（Hash），以防資料庫外洩時 Token 被直接複用。

### 6.8 Session Token 有效期與續期策略

* **有效期限**：Session Token 預設有效期為 **8 小時**（可於環境變數中配置）。
* **續期機制（Sliding Window）**：使用者在 Token 有效期內持續操作時，後端在每次 API 請求成功後自動將 Token 有效期順延（重置為 8 小時），無需使用者重新登入。
  * **續期併發防護 (Concurrency Protection)**：在執行 Session 續期寫入時，若遭遇背景垃圾回收 (GC) 或多視窗併發登出導致 Session 已被提前刪除的競態條件 (Race Condition)，系統必須妥善捕捉資料庫之髒資料例外（如 `StaleDataError`）並安全地 Rollback，將其視為無效操作，絕對避免造成伺服器 500 例外崩潰。
* **強制過期**：以下情況強制使 Token 立即失效，不論是否在有效期內：
  * 使用者主動登出。
  * 管理員停用（Suspend）該帳號。
  * Token 已超過最大絕對有效期（如 7 天，不論是否持續操作）。
* **同時登入**：系統允許同一帳號在多個裝置上同時持有有效 Session（`sessions` 表中可存在多筆有效記錄），各 Session 獨立管理有效期。

### 6.9 初始管理員帳號 Bootstrap 機制

邀請制系統中，第一個管理員帳號須透過以下機制建立，而非透過前台或後台介面：

* **CLI 指令**：提供專用 CLI 指令 `python cli.py --create-admin <email>`，執行後在 Auth DB 中直接建立一個狀態為 `Active`、角色為 `Admin` 的帳號，並在終端機輸出系統隨機產生的高強度初始密碼。管理員以此隨機密碼首次登入時，系統會強制攔截並要求其完成「強制設定密碼」流程，徹底落實零信任與密碼不落地原則。
* **安全限制**：該 CLI 指令僅允許在伺服器本機端執行（不透過 API 暴露），且執行時需確認 Auth DB 尚未存在任何 Admin 帳號（防止重複建立）。
* **後續管理員新增**：第一位管理員建立後，後續管理員帳號統一透過後台「角色管理」功能指派，不再使用 CLI 指令。

## 7. 前台功能需求 (Frontend Functional Requirements)

前台面向一般受邀使用者，提供任務操作、結果查閱等核心功能。

### 7.1 身分驗證與帳號

* **登入頁面**：支援以電子郵件 + 密碼登入；首次登入可接受以電子郵件 + 邀請 UUID 登入。
* **首次登入密碼設定頁面**：如 §6.2 所述，強制於功能頁面前完成密碼設定。介面需提供**即時密碼強度指示器 (Password Strength Indicator)**，動態給予使用者複雜度達標的視覺反饋。
* **帳號設定頁面**：使用者可修改個人密碼（需先驗證現有密碼）。
* **登出功能**：提供登出按鈕，清除 Session 並返回登入頁面。

### 7.2 任務管理（前台）

* **任務列表頁面**：
  * 顯示當前使用者的所有任務，含任務名稱、起始 URL、建立時間、狀態（等待中、執行中、暫停、完成、異常）及進度摘要。
  * 支援依狀態篩選、依建立時間排序。
* **任務識別碼 (UUID) 呈現與檢視規範**：
  * **完整 UUID 呈現**：在「我的任務列表」表格中，任務 ID 必須完整顯示其原始原生 UUID（保留 36 個字元的完整格式，不進行縮小或截減前 8 碼），且列表的篩選功能必須支援使用完整 UUID 或是其部分字串進行比對搜尋。
  * **選單與列表格式化優化**：在「複製任務」、「任務比對」與「移交任務」等具有下拉選單或緊湊列表的 UI 視窗中，任務 ID 則統一採用 `ABCDE...FGHIJ` 格式化規則（保留前 5 碼與後 5 碼，中間以 `...` 串接），以在有限空間內保持版面美觀且利於識別。
* **新增任務**：
  * 表單輸入起始 URL、最大深度、最大頁面數、延遲設定等參數。
  * 前端即時驗證 URL 格式合法性。
  * **自動網域防呆填寫 (Domain Auto-Fill Assist)**：為提升使用者體驗並減少人為輸入失誤，當使用者在表單輸入起點網址（Start URL）並離開焦點（blur）時，前端必須自動解析其網域主機名稱（Hostname），並預先填入「目標網域」與「信任網域」輸入框中。
  * **複製任務之表單精準回填 (Form Data Difference Filtering)**：在執行「複製任務」時，前端必須針對陣列與字典型態的參數（如忽略的副檔名、特定網域延遲等）實作「差集運算 (Difference)」。必須嚴格過濾掉已存在於全域預設中的項目，確保新建表單僅精準回填使用者的「自訂覆寫值」，防止全域設定被硬編碼為區域任務設定。
  * 系統需動態向後端請求全域預設配置與安全上下限，並自動填入表單作為預設值。
* **任務詳情頁面**：
  * 提供獨立的檢視對話框，顯示任務建立當時被鎖定的「執行配置快照（格式化排版）」。
  * 顯示執行進度：已爬頁面數 / 佇列總數、目前深度層級、已發現外連數量。
  * 提供操作按鈕：暫停、恢復、重置、刪除。不僅針對執行中的任務，針對處於 `queued` 或 `starting` 狀態的任務也必須開放「暫停」權限，允許使用者在排隊期間及時攔截並中止任務。
  * **即時狀態更新 (Server-Sent Events)**：任務執行中，前台應透過 Server-Sent Events (SSE) 串流即時反映任務執行進度。後端 SSE 端點於任務狀態發生變化時推送更新事件，並在任務進入終態（`completed`、`error`）或非執行狀態（`paused`、`pending` 且進程未運行）時自動關閉串流。前端在初始載入時，若偵測到任務正在執行中，應自動建立 SSE 連線；當串流關閉或發生錯誤時，前端需優雅處理並停止連線，避免資源洩漏。

### 7.3 結果查閱與匯出

* **嚴格的 API 分頁與聚合查詢契約 (Strict Pagination & Aggregation)**：
  * 前端在串接外連結果查詢 API（`GET /api/jobs/{job_id}/results`）時，必須嚴格遵守參數傳遞契約，不依賴後端模糊的預設值。
  * 必須明確傳遞分頁參數 (`page`, `page_size`) 與聚合狀態 (`group_by`)。即使處於不進行任何聚合的原始檢視模式，也必須強制帶入 `group_by=none`，以防範後端查詢邏輯因參數缺失產生歧義或觸發伺服器 500 例外。
* **外連結果列表**：
  * 以表格顯示所有已探索的外部連結，欄位必須包含：來源頁面、目標頁面、IP 位址、HTTPS（為避免語意歧義與錯誤安全感，嚴禁命名為「安全」）、HTTP 狀態碼與錯誤訊息。
  * **欄位順序統一原則**：為保持閱讀直覺與追溯一致性，全站所有報表（包含前端外部/內部連結表格，以及 CSV / ZIP 大禮包匯出檔案），其欄位排列皆必須嚴格統一遵循「**來源頁面 -> 目標頁面**」之邏輯順序。
  * 支援透過上方統計卡片依外連狀態與風險等級進行篩選：正常連結、DNS 錯誤、**實質失效**（高風險：如 404/500 等伺服器錯誤，若非防爬蟲措施則屬高風險）、**權限阻擋**（低風險：遭 WAF 阻擋如 401/403/429）與非 HTTPS 連結。
  * 支援依目標網域、來源頁面關鍵字搜尋。
  * **外部連結重新探測 (External Reprobe)**：提供針對外部連結重新探測之功能機制。在「依目標頁面聚合」分組模式下，可勾選該外連目標執行重測，其重新探測的標的為**「選定的外部 URL 本身」**（重新發送 HTTP 請求以確認該外部網站是否恢復正常）；在「依來源頁面聚合」模式下，則比照內部連結模式處理，其重新探測的標的為**「包含這些外部連結的內部來源網頁 (Source URL)」**（這意味著爬蟲會重新發起請求去抓取該內部網頁，並刪除該網頁於本次任務衍生之所有外部連結，以確認網頁原始碼中的失效連結是否已被移除或修正）。此外，為避免無效勾選與業務邏輯衝突，在「平舖列表 (未聚合)」與「依外部網域統計」這兩種模式下，介面必須隱藏並禁用重新探測按鈕及勾選框。特別注意，對於「非 HTTPS」之錯誤分類，因其問題根源為來源頁面之網址寫法而非目標頁面異常，故在「依目標頁面聚合」模式下若過濾條件包含非 HTTPS，介面亦必須強制隱藏並禁用重新探測功能。
  * **去重聚合與動態摘要**：支援多維度聚合模式 (Target/Source/Domain) 合併與統計外連數據。當切換聚合模式或排除特定網域時，上方的統計卡片必須動態依據該聚合維度重新計算與顯示精確的數量。
  * **排除網域與狀態記憶**：支援於前端檢視與匯出時自訂「排除網域清單」，並透過瀏覽器本地儲存 (`localStorage`) 自動記憶。提供「啟用/停用」排除過濾的快速切換開關，提升多情境維運時的便利性。
* **內部連結診斷與精細化樣態分類 (Internal Link Diagnostics & Categorization)**：
  * 系統需提供獨立頁籤顯示目標網站的內部連結診斷。為協助不同維運角色快速定位與修補，系統必須實作「8 大核心失效樣態分類」與統計儀表板：
    1. **伺服器異常 (Server Error)**：伺服器無法完成請求 (5xx)（通常由後端工程師或 IT 處理）。
    2. **底層異常 (Connection Error)**：憑證無效、DNS 異常或連線被拒，無狀態碼且非逾時錯誤。
    3. **連線逾時 (Timeout)**：伺服器無法於時限內回復，無狀態碼且錯誤訊息包含 `timeout` 或 `timed out`（通常需進行效能優化）。
    4. **資源遺失 (Not Found)**：資源請求失敗 (404) 或資源已刪除 (410)（通常由內容編輯處理）。
    5. **其他異常 (Other Error)**：未預期的例外，不屬於上述之其他 HTTP 異常狀態（如 400, 405 等防呆捕捉）。
    6. **網頁截斷 (Truncated)**：網頁容量超過最大限制被提早截斷，可能遺漏部分外部連結。
    7. **權限不足 (Access Denied)**：未獲授權 (401) 或拒絕進入 (403)（極可能是正常防爬蟲行為，因此置於最後）。
    8. **非 HTTPS (Insecure)**：使用了未加密的 HTTP 協定進行連線，容易遭到中間人監聽或劫持（不論 HTTP 請求是否成功皆會被捕捉與標記）。
  * **內部網頁重新探測 (Internal Reprobe)**：在未分組的「平舖列表」檢視下，支援勾選特定內部網頁並執行「重新探測」。其重新探測的標的為**「該選定的內部網頁 (Internal URL) 本身」**。此操作會將該內部網址狀態重設為等待中，要求爬蟲重新抓取與解析該頁面，且同時刪除該網址先前於此任務中衍生的所有外部連結紀錄，確保重新爬取時不會產生重複髒資料。為確保一致性，在「依來源頁面聚合」以外之分組模式下將隱藏此功能。特別注意，對於「非 HTTPS」之錯誤分類，因其問題根源為來源頁面之網址寫法，在「平舖列表」模式下若過濾條件包含非 HTTPS，介面亦必須強制隱藏並禁用重新探測功能。
  * 介面上方需以視覺化統計卡片並排呈現各類別數量（總計文案應標示為「包含失敗與截斷的總數量」），並支援點擊卡片快速篩選下方表格資料。當卡片處於篩選啟用 (Active) 狀態時，必須提供明確的視覺高光回饋（如提亮邊框與加深光暈陰影）。
  * 表格需清楚列出來源頁面、失效網址、HTTP 狀態碼（依嚴重性套用不同顏色識別）與錯誤訊息；支援「依來源頁面聚合 (修補視角)」且統計卡片同步連動，並提供專屬的 CSV / JSON 完整報表匯出功能。
* **任務歷史差異比對引擎 (Job Diff Engine)**：
  * 系統已實作強大的差異比對引擎，支援選定兩個歷史任務進行比對，作為長期追蹤網站外連資安健康度的重要維運亮點。
  * 後端比對引擎具備六大維度分析能力：**IP 發生異動**（防範網域搶註與 SSRF 劫持）、**健康度劣化**（連結失效）、**安全降級**（HTTPS 降為 HTTP）、**狀態復原**，以及**新增/消失外連**。
  * 前端需以視覺化頁籤呈現差異報表，並彙整標示所有受影響的來源母網頁 (Source URLs)，以利開發人員快速溯源修補。
  * 針對「IP 發生異動」維度，系統必須以「網域 (Domain)」為單位進行聚合顯示，並統計受影響的 URL 總數，避免巨量重複資訊洗版。
* **結果匯出**：
  * 支援將篩選後的結果匯出為 CSV 或 JSON 格式下載。
  * 支援完整報表 (ZIP 壓縮檔) 下載功能。
  * 匯出內容需與 CLI 匯出功能保持一致（欄位對齊）。
* **任務報告摘要**：提供任務維度的統計摘要，包含總爬取頁數、外連總數、損毀連結數、非白名單數、HTTP 異常數，以及內部連結失效總數（涵蓋各失效樣態分類數量）等。
* **前台結果檢視防護與效能優化（Frontend Throttling & Performance Optimization）**：為因應使用者頻繁變更篩選、排序、分組或卡片點選時所產生的頻繁資料拉取，系統前台必須實作防護與優化機制：
  * **篩選防抖 (Filter Debounce)**：表格欄位的過濾輸入框（Filter Inputs）在輸入時必須實作 500ms 的防抖延遲（Debounce），避免使用者在打字時對後端伺服器發送大量冗餘的 HTTP 請求。
  * **半透明載入緩衝 (Opacity Loading Buffer)**：當使用者異動篩選、排序、分組且原本已有渲染表格時，系統不應清空整個區塊，而應將現有表格透明度設為 0.5，作為流暢且低干擾的載入中視覺反饋，避免畫面全白閃爍。
  * **視圖切換重置防閃爍 (View Switch Reset)**：在切換到「任務詳情」等需要依賴非同步 API 取得資料的頁面時，前端必須「先清空舊任務的所有畫面內容」（包含統計卡片、表格資料、狀態標籤等），並顯示明確的載入中狀態。嚴禁在等待新任務 API 回應期間，畫面上殘留前一個任務的舊資料，以免造成嚴重的資訊混淆。
  * **頁籤焦點延遲載入 (Tab-Focused Lazy Loading)**：在包含多個資料維度（如外部連結與內部連結）的介面中，前端必須根據當下啟用的「頁籤 (Tab)」動態決定載入對象。嚴禁在載入畫面時一次性發送所有頁籤的 API 請求，未顯示的頁籤必須延遲至使用者實際點擊切換時才觸發資料載入，藉此大幅節省伺服器運算與資料庫資源。
  * **API 串行加載減壓 (Sequential Loading)**：重新載入結果時，系統必須採用串行加載機制。優先加載並渲染分頁表格資料，待表格加載完成後，才發送請求去拉取統計卡片摘要（Summary），以此分散並減輕後端大運算並發請求的 CPU 與 IO 負載。
  * **Summary 統計 API 快取機制 (Summary Cache)**：統計摘要 API（`/results/summary` 與 `/internal-results/summary`）的查詢成本高昂（涉及大量 `GROUP BY` 與 `COUNT` 運算）。由於切換統計卡片（Filter Card）、翻頁或進行欄位搜尋並不影響各分類的數量統計，系統前端必須實作 Summary 快取機制：
    * **快取判斷**：在呼叫 Summary API 前，先比對本次請求的「決定性參數」（外連：`exclude` + `group_by`；內連：`group_by`）是否與上次已快取的結果完全吻合。若吻合，則直接使用快取渲染統計卡片，不再對後端發出任何請求。
    * **強制失效時機**：下列情況下必須清除快取，迫使下次強制重新請求：切換至不同任務、排除網域（exclude）設定變更、聚合模式（group\_by）變更、任務執行狀態由 `running`/`starting` 轉為其他狀態（SSE 驅動）、使用者執行「重新探測」、「重置任務」或「重試失敗項目」等導致資料實際變動的操作。
    * **不觸發失效的操作（得以省去 API 請求）**：在統計卡片（Filter Card）之間切換、翻頁（分頁切換）、表格欄位排序與欄位搜尋、Tab 切換回同一 Tab（`group_by` 與 `exclude` 未變時）。
    * **活躍任務之節流輪詢 (Throttled Polling)**：當任務處於 `running` 或 `starting` 狀態且使用者正停留在該任務的詳情頁時，為兼顧即時數據體驗與資料庫效能，前端必須啟動定時器 (`setInterval`)，每隔 30 秒自動向後端拉取一次最新版本的 Summary 資料來刷新畫面上的統計卡片。為避免干擾使用者閱讀與操作，此定時輪詢**絕對不可**觸發結果清單列表 (`/results`) 的重新載入。當任務轉入終態、暫停，或使用者離開詳情頁時，必須立刻清除該輪詢定時器以釋放資源。

* **後端查詢與資料庫效能優化 (Backend Query & Database Performance Optimization)**：
  * **規避 PostgreSQL `ORDER BY ... LIMIT` 全表掃描陷阱**：在實作分頁查詢時，若遭遇稀疏條件過濾（例如只篩選數百萬筆資料中極少數的失效連結）並搭配 `ORDER BY id LIMIT n`，PostgreSQL 查詢最佳化器極易發生誤判，為了節省排序成本而選擇循序掃描 Primary Key 索引，最終導致災難性的全表掃描。為規避此問題，後端在動態排序時必須將預設的排序欄位轉換為表達式（例如：`default_sort = id + 0`），強制破壞該欄位的純粹性使資料庫無法依賴主鍵索引排序，進而被迫使用建立好的 Partial Index (部分索引) 先把少數符合條件的資料撈出後再於記憶體排序，確保查詢在毫秒內極速完成。此 `+ 0` 技巧亦完全相容於 SQLite 等輕量資料庫。
  * **嚴謹的 SQLAlchemy Boolean 條件對應**：在後端 ORM 查詢中，針對 Boolean 條件（如 `is_secure`），嚴禁使用 `.is_(False)`，因為這會產生 `IS false` 的 SQL 語法，這在 PostgreSQL 中無法精確吻合建立部分索引時所宣告的 `is_secure = false` 條件，導致優化器放棄使用索引。必須強制使用 `== False` 來精準產生 `= false` 語句，確保與 Partial Index 的抽象語法樹 (AST) 完美契合。

### 7.4 全站資料表互動 (Client-side Table Sorting & Filtering)

* **全域過濾與排序**：全站所有的主要數據表格（包含「我的任務列表」、「任務詳情中的外部連結診斷與內部失效連結」、「使用者管理列表」、「任務監控列表」以及「操作日誌列表」）必須強制統一採用客戶端高效能資料表引擎，全面支援點擊表頭欄位進行排序（▲/▼/⇅），以及在各個欄位下方提供即時輸入篩選框（Filter Inputs），以大幅降低後端 API 的查詢負載，並提供流暢的數據交互與分析體驗。
* **批次勾選與操作支援 (Batch Selection & Operations)**：內外部連結診斷等具有操作需求之資料表格，必須實作前端 Checkbox 批次勾選功能，允許使用者跨頁保留選取狀態，以利針對特定多筆連結進行局部的「批次重新探測」或「局部匯出」。

### 7.5 前端狀態管理與元件封裝防呆規範

為避免純 Vanilla JS 專案隨著功能擴充而導致「義大利麵條式程式碼 (Spaghetti Code)」與狀態混亂，前端開發必須遵循以下規範：
* **Web Components 與類別封裝**：UI 獨立區塊（如任務卡片、資料表格、統計面板、警告吐司框等）必須封裝為獨立的類別 (Class) 或原生 Web Components (Custom Elements)。每個元件必須自行封閉管理其內部的 DOM 操作與狀態，對外僅暴露必要的建構子與更新介面。
* **單一職責與副作用隔離 (Separation of Side Effects)**：前端在載入或初始化視圖（如詳情頁載入）時，必須維持純粹的資料渲染職責。嚴禁在元件初始化邏輯中夾帶會「重複觸發其他 API 拉取」的副作用，所有依賴關係（如任務重測後應更新列表）必須改為由明確的事件或使用者互動操作來統一調度，防範 API 的多重重複請求風暴。
* **全域狀態防護**：嚴格限制全域變數的使用，各頁面的狀態必須封裝在該頁面的主要 Controller 或 Manager 類別中，確保多個元件間的狀態流轉具備單向性與可追蹤性。

### 7.6 使用者介面體驗

* 全站操作需具備清晰的錯誤提示（如 URL 格式錯誤、無權限等）。
* **公用 UI 封裝與 Toast 非阻塞通知系統 (Toast Notification System)**：全站操作反饋（如設定儲存、操作成功、網路連線錯誤等）必須採用優雅的非阻塞 Toast 通知系統，統一於畫面角落彈出並支援自動淡出，嚴禁使用會阻塞使用者操作的瀏覽器原生 `alert`。前端公用元件必須高度封裝，以確保 UI 體驗一致性。
* 長時間操作（如任務執行中）需提供明確的載入指示器（Progress Indicator）。
* 介面設計需具備基本的響應式佈局（RWD），支援主流桌面瀏覽器。
* **海量資料分頁導覽強化 (Pagination UX)**：全站所有具備分頁功能的數據表格（包含內/外連診斷報表、後台操作日誌等），其分頁元件必須提供「第一頁 («)」與「最後一頁 (»)」之快速跳轉按鈕，並附帶明確的滑鼠懸停提示 (Tooltip)，大幅減少使用者在數百頁資料間的反覆點擊疲勞。
* **對話框防呆與滾動穿透防禦 (Modal Dialog & Scroll Bleed Defense)**：
  * **對話框防呆**：嚴禁使用瀏覽器原生之 `alert`、`confirm`、`prompt`，所有破壞性操作（如刪除、停用、重置）必須使用自訂之 Modal 視窗進行二次確認，並依照危險等級套用警示色彩。
  * **滾動穿透防禦 (Scroll Bleed Prevention)**：為提供高品質的彈窗操作體驗，當全站任何 Modal 彈出視窗（如設定參數檢視、修改密碼、排除網域、刪除確認等對話框）處於開啟狀態時，系統必須自動且徹底鎖定背景網頁 `body` 的滾動行為（如透過 `body.modal-open` 動態套用 `overflow: hidden !important`）。同時，對所有 Modal 遮罩層與內容容器套用 `overscroll-behavior: contain` 屬性，阻斷滾動事件鏈向下傳遞，解決滾動穿透問題。
* **統計與進度卡片自適應與均分配置 (Responsive Stats Cards)**：
  * **自適應換行佈局**：任務詳情的各項統計卡片區域（如「爬取進度」、「外部連結診斷」、「內部連結診斷」與比對結果等），必須統一採用 CSS Grid 的 `auto-fit` 自動換行佈局（如 `repeat(auto-fit, minmax(100px, 1fr))`），確保在窄螢幕或空間不足時，卡片能優雅換行並保持最佳的閱讀尺寸。
  * **標籤對稱性**：為保持版面精簡對稱與美觀，爬取進度的 6 個卡片標籤必須統一限制為 2 個字（總計、完成、截斷、等待、略過、失敗），防範長詞語導致版面擁擠。
  * **數字字型自適應縮放 (CSS Container Queries)**：卡片內部的數值字型大小必須使用 CSS 容器查詢技術，隨著卡片本身的實際寬度動態調整（使用 `clamp` 配合 `cqi` 單位，如進度卡片採用 `clamp(1rem, 28cqi, 2.25rem)`），在空間寬裕時呈現醒目大字體，在空間受限時自動縮小，且**絕對嚴禁截斷數值或以 `...` 隱藏**，必須完整呈現所有數值。
* **非同步資料競爭防禦 (Async Data Race Prevention)**：在進入任務詳情等透過非同步 API 載入資料的頁面，或在極短時間內頻繁切換頁籤與過濾條件時，若前一個未完成的 API 請求在操作切換後才回應，極易造成舊資料覆蓋新畫面的狀態競爭 (Race Condition)。因此，前端必須實作嚴謹的雙重防禦機制：在任何非同步 API 回傳並更新 DOM 之前，必須嚴格比對回傳資料所屬的「任務識別碼 (Job ID)」與「遞增請求序號 (Request ID)」是否與當下畫面狀態完全吻合。若有任一不符則必須直接捨棄該次回應，徹底杜絕畫面殘留舊任務或舊篩選狀態的異常情況。

## 8. 後台功能需求 (Backend Admin Functional Requirements)

後台面向系統管理員，提供使用者管理、全域配置、系統監控等功能，存取需具備管理員角色（Admin Role）。

### 8.1 管理員身分驗證

* 管理員使用獨立帳號登入後台，或由角色權限區分（RBAC）控管存取。
* 後台路徑需與前台路徑明確隔離，未授權存取一律返回 403 或重導向至登入頁面。

### 8.2 使用者管理

* **使用者列表**：
  * 顯示所有帳號，含電子郵件、帳號狀態、建立時間、最後登入時間、角色。
  * 支援依帳號狀態篩選（Pending / Active / Suspended / Expired）。
* **新增使用者（邀請）**：
  * 輸入受邀者的電子郵件信箱。
  * 系統產生 UUID、設定有效期限，並寄出邀請信。
  * 可配置邀請有效期限（預設 72 小時）。
* **重新寄送邀請**：針對 Pending 或 Expired 狀態的帳號，產生新的 UUID 並重新寄出邀請信，舊 UUID 立即失效。
* **停用 / 啟用帳號**：管理員可切換帳號為 Suspended / Active 狀態，停用後該帳號立即無法登入。
* **刪除帳號**：管理員可永久刪除帳號及其所有關聯任務資料（需二次確認）。
* **角色管理**：支援將一般帳號提權為管理員（Admin），或將管理員降權為一般使用者（User）。

### 8.3 任務監控（後台）

* **全任務列表**：顯示全部使用者的所有任務（不限於當前管理員），可依使用者、狀態篩選。
* **任務維運與強制操作**：管理員可對任意任務執行以下操作：
  * **暫停 / 強制接管**：將任意執行中的任務強制暫停。
  * **強制取回 (Force Retrieve)**：將其他使用者的任務轉移給自己（管理員）。實作上需注意：**針對本身即為管理員自己擁有的任務，介面上不應顯示此按鈕，避免混淆**。
  * **任務刪除**：強制刪除指定任務與其關聯資料。
* **任務資料備份與遷移**：提供 GUI 介面的資料匯出與匯入功能（底層對接 `job_sync.sh` 的邏輯）。
  * **匯出 (Export Backup)**：允許管理員針對單一任務，將所有相關資料匯出為 ZIP 備份檔。
  * **匯入 (Import Backup)**：允許管理員上傳 ZIP 備份檔並還原為系統任務。實作規範：
    * 遵循 Content Security Policy (CSP) 規範，不得在 HTML 中使用 Inline Event Handlers (如 `onclick`)，須以獨立的 JavaScript 監聽事件實作。
    * 必須具備上傳進度的防呆機制：上傳期間按鈕應處於鎖定狀態並顯示「匯入中...」。
    * 必須防止意外中斷：上傳期間若使用者嘗試切換頁面或關閉視窗，需透過 `beforeunload` 與 Web Component 攔截機制（例如 `<confirm-modal>`）跳出警告，避免傳輸異常。
* **執行狀態概覽**：提供全系統任務執行概覽，包含目前執行中任務數、佇列等待中任務數、異常任務數等。

### 8.4 全域配置管理

* 提供 Web 介面檢視與修改系統全域配置，包含：
  * 爬蟲安全上下限（逾時、TCP 連線逾時、探測逾時、延遲、重試次數、最大深度與最大頁數）
  * 全域忽略副檔名清單（`ignore_extensions`）
  * 全域忽略路徑規則（`ignore_regexes`）
  * 自簽憑證豁免網域（`ssl_exempt_domains`）
* **配置關聯性驗證**：後端 API 在接收配置更新時，必須對安全上下限配置進行交叉驗證（如 `min_timeout` 不得大於 `max_timeout`），防止邏輯矛盾導致爬蟲引擎啟動失敗。
* 配置修改需寫入安全操作日誌（`AuthLog`），詳細記錄修改的操作者 ID、時間、IP 位址以及修改前與修改後的配置內容差異（`detail` 欄位以 JSON 儲存）。

### 8.5 郵件配置管理

* 後台提供郵件伺服器（SMTP）設定狀態之唯讀檢視介面（主機、埠號、帳號、寄件者名稱等）。
* 基於安全考量與機密隔離原則，SMTP 配置嚴禁透過 Web UI 修改與儲存於資料庫，必須統一自伺服器環境變數載入。
* 密碼欄位在後端 API 回傳與前端顯示時必須進行遮罩（Masking）處理，絕對不得回傳明文。
* 支援「寄送測試郵件」功能，以驗證當下環境變數之 SMTP 配置是否正常運作。

### 8.6 系統日誌查閱

* 後台提供系統操作日誌查閱介面，涵蓋儲存於 `auth_logs` 的各類事件，包含：
  * 使用者登入 / 登出 / 鎖定等紀錄（含 IP、時間戳）
  * 帳號邀請與狀態變更紀錄（如停用、變更角色）與刪除帳號紀錄
  * 全域配置變更紀錄（記錄修改前後的詳細欄位差異）
  * 任務強制操作紀錄（如強制暫停 `takeover`、強制刪除 `delete` 等）
* 日誌支援依時間範圍（API 需支援 `start_date` 與 `end_date` 參數以 ISO 8601 或 YYYY-MM-DD 格式進行精確的日誌時間區間過濾）、事件類型（`event_type`）與操作使用者（`user_id`）進行篩選。

### 8.7 後台技術實作規範

後台（Admin）介面與前台共用同一後端 API 服務，以角色權限（RBAC）控管存取，**不另建獨立後台應用**。後台前端頁面同樣遵循 §9.1 所訂之 Vanilla JS + ESM 技術規範：

* **後台路由隔離**：後台相關頁面路由統一以 `/admin/` 前綴區分，後端中介層（Middleware）強制驗證請求方是否具備 Admin 角色；缺乏權限時一律回傳 403，不洩漏任何後台資源。
* **操作確認保護**：高風險操作（刪除帳號、強制接管任務、重置配置）需在後台前端呈現二次確認對話框，並於後端 API 端再次驗證操作者角色，防止 CSRF 或誤觸。
* **配置頁面輸入驗證**：全域配置修改表單須於前端進行格式預驗（如正規運算式語法、數值範圍），並於後端 API 進行最終合法性校驗，兩者缺一不可。

## 9. 系統技術堆疊與實作規範 (Technology Stack & Implementation Specs)

本節定義整個系統（前台、後台、API、爬蟲核心）的技術選型原則，作為後續實作的最高指導依據。

### 9.1 前台技術堆疊

前台介面採**輕量原生技術棧**，以降低供應鏈風險、減少打包複雜度與長期維護成本：

* **JavaScript**：
  * 強制使用 **Vanilla JS**，禁止引入 React、Vue、Angular、Svelte 等前端框架。
  * 模組化採用瀏覽器原生 **ES Modules（ESM）**（`<script type="module">`），允許使用 `import` / `export` 進行程式碼拆分，無需打包工具（Bundler）。
  * 禁止使用 jQuery 或其他 DOM 操作函式庫。
  * **時區與時間本地格式化規範**：
    * 前端顯示的所有時間戳記（如任務建立時間、更新時間、登入時間、日誌時間等）皆必須依據使用者的**瀏覽器當地時區**進行自動動態調整。
    * 所有日期與時間的顯示格式必須強制統一為 `YYYY/MM/DD HH:mm:ss`（採用 24 小時制，且年、月、日、時、分、秒皆須進行補零對齊，如 `2026/06/13 14:09:52`），以確保全站時間顯示的一致性。
  * **XSS 防禦與 DOM 渲染規範**：為防範跨網站指令碼 (XSS) 攻擊並符合嚴格的 CSP 規範，**前端 JavaScript 程式碼全面禁用 `innerHTML` 屬性**（包含靜態版型的建構）。必須強制使用 `document.createElement` 與 `textContent` 賦值。若遇特殊情況需輸出文字，亦應透過自訂的 `escapeHtml()` 工具函式對來源資料進行 HTML 實體跳脫處理。
  * **API 請求統一封裝**：前端所有與後端的資料交換必須統一透過封裝後的 `fetch` 工具函式發送。該模組需負責自動讀取並夾帶 `X-CSRF-Token` 標頭，並統一攔截處理 HTTP 401 (Unauthorized) 回應以執行自動重導向至登入頁面之邏輯，確保全站驗證狀態之一致性。
  * **原生單頁應用路由 (Native SPA Routing)**：系統需實作基於 `hashchange` 事件的純前端路由機制（Hash-based Routing）。在切換視圖（如任務列表、任務詳情）時，只需動態替換 DOM 內容或切換顯示狀態，嚴禁觸發整頁刷新 (Page Reload)，以提供流暢的使用者體驗。
* **CSS**：
  * 使用 **Vanilla CSS**（含 CSS Custom Properties / CSS Variables），禁止引入 Tailwind CSS、Bootstrap 等 CSS 框架。
  * 允許使用 CSS 原生 `@import` 或 CSS Modules（若伺服器端支援）進行樣式分拆。
* **第三方套件原則**：
  * 原則上**不引入第三方前端套件**；若確有必要（如圖表渲染），須經評估後於需求文件中明確列出，並鎖定版本與來源（使用 SRI Hash 驗證完整性）。
  * 禁止透過公共 CDN 動態載入未鎖定版本的第三方資源。
* **後端整合**：
  * 前台與後端透過 **REST API（JSON）** 溝通，前台不直接渲染伺服器端模板（Server-Side Template）。
  * HTTP-only Cookie 承載 Session Token。

### 9.2 後端技術堆疊（Python Web 服務）

* **語言**：Python 3.11+。
* **Web 框架**：採用輕量 Python Web 框架（如 Flask 或 FastAPI）；選型需確保能與現有爬蟲核心模組整合，且支援非同步 I/O（用於任務狀態推送）。
* **API 風格**：RESTful JSON API；所有端點需明確定義 HTTP 方法語義（GET / POST / PATCH / DELETE）。
* **身分驗證**：使用 HTTP-only Cookie 承載 Session Token（Token 值為 Auth DB 中 `sessions.token_hash` 的對應原始值）；禁止將 Token 儲存於 `localStorage` 或 `sessionStorage`。
* **健康檢查端點 (Health Check)**：系統必須提供 `/api/health` 輕量檢查端點，回傳 HTTP 200 狀態碼以供外部負載平衡器 (Load Balancer)、Docker 容器或 CI/CD 服務監控系統存活狀況。
* **靜態資源快取 (In-Memory Cache)**：在生產環境模式下，針對頻繁存取的靜態 HTML 頁面（需動態注入 CSP Nonce），後端應實作記憶體字典快取機制，消弭不必要的磁碟讀取 I/O 開銷。
* **暫存檔案生命週期管理 (Temp File GC)**：在處理匯出完整報表 (ZIP 壓縮檔) 等需依賴實體磁碟暫存檔的操作時，後端服務必須強制利用非同步背景任務 (如 FastAPI `BackgroundTasks`)，確保 HTTP 回應傳輸完畢或發生異常中斷後，自動且確實地清理伺服器上的實體暫存檔，防範長期運行下的磁碟空間洩漏 (Disk Space Leak)。
* **前端路由容錯導向 (SPA Fallback)**：為配合前端原生單頁應用程式 (SPA) 的路由機制，Web 服務層需攔截非 API 或靜態資源的 HTTP 404 請求，將其自動重導向至系統首頁 (`/`)，防止使用者因網址輸入錯誤或重新整理而遭遇中斷體驗。
* **高併發與非同步事件迴圈保護 (Event Loop Protection)**：系統後端採用 ASGI 框架 (FastAPI)，必須嚴格遵守非同步開發規範。所有包含同步阻塞操作（如 SQLAlchemy 資料庫 I/O、`bcrypt` 密集 CPU 運算、SMTP 同步網路請求）的 API 端點，必須宣告為標準同步函式 (`def`)，交由底層執行緒池處理，絕對禁止使用 `async def` 以免阻塞主事件迴圈，確保系統能承受企業級高併發連線。

### 9.3 資料庫

* **引擎選型與環境隔離**：
  * **Auth DB**：建議使用 SQLite（開發）或 PostgreSQL / MySQL（生產），以支援交易與行級鎖定。系統必須支援透過 `.env` 檔案中的 `AUTH_DB_URL` 環境變數自訂連線路徑。
  * **Crawler DB**：延續現有爬蟲資料庫引擎（SQLite）。系統必須支援透過 `.env` 檔案中的 `CRAWLER_DB_URL` 環境變數動態抽換路徑，以落實開發、測試、生產環境之物理隔離。
  * 針對 SQLite，為配合 FastAPI/Starlette 的 `StreamingResponse` 跨執行緒池串流輸出，必須設定 `check_same_thread=False`，並搭配 `WAL` 模式以保障高併發讀取與生成器跨執行緒操作的安全。
* **Schema 遷移**：使用遷移工具（如 Alembic）管理兩個資料庫的 Schema 版本，遷移腳本需納入版本控制並可在 CI/CD 中自動執行。
* **ORM / 查詢**：可選用輕量 ORM（如 SQLAlchemy Core）或直接使用參數化 SQL 查詢；**嚴禁字串拼接 SQL**，以防 SQL Injection。

### 9.4 郵件發送

* 使用 Python 標準 `smtplib` 或輕量封裝（如 `aiosmtplib`）發送郵件，**不引入完整的第三方郵件服務 SDK**（如 SendGrid、Mailgun）以維持自主控制。
* SMTP 連線需支援 STARTTLS 或 SSL/TLS；禁止使用明文 SMTP 連線。
* 郵件範本以純文字或最小化 HTML 撰寫，禁止在郵件中嵌入外部追蹤像素或外連資源。
* **開發模擬模式 (Console Mode)**：為支援本地開發與無網路環境測試，系統需支援透過設定 `SMTP_CONSOLE_MODE` 切換至模擬模式。啟用時，不實際發起 SMTP 網路連線，而是將郵件內容（含邀請連結與密碼）安全輸出至伺服器主控台日誌，降低開發測試障礙。

### 9.5 部署與環境

* **環境區隔**：開發（dev）、測試（test）、生產（prod）環境的配置需完全分離，透過環境變數注入差異設定。
* **機密管理**：資料庫路徑 / DSN、SMTP 密碼等機密值**嚴禁硬編碼於程式碼或版本控制中**，一律透過環境變數或受保護的 Secret 管理工具提供。
* **依賴鎖定**：前後端所有依賴套件需明確鎖定版本（Python 端使用 `requirements.txt` 或 `pyproject.toml` + lock 檔），確保部署環境可重現。
* **HTTPS**：生產環境前端與 API 服務一律透過 HTTPS 提供，HTTP 請求須強制重導向至 HTTPS；禁止在生產環境關閉 SSL/TLS 憑證校驗。

## 10. 效能與穩定性 (Performance & Stability)

### 10.1 記憶體保護與巨量資料串流處理

* **OOM Defense & Data Streaming**：系統在處理所有可能包含數十萬筆紀錄的資料表（如 `crawl_queue`, `external_links`）時，全面採用 SQLAlchemy 的 `yield_per()` 批次迭代技術。無論是進行資料庫遷移、任務備份匯出、或是產生線上報表，皆以固定大小的區塊 (Chunk) 串流讀取，確保即使面對巨量資料，伺服器的記憶體消耗依然維持在極低水平，徹底根絕 OOM (Out of Memory) 崩潰風險。
* **巨量報表串流匯出 (Streaming Export)**：針對數十萬筆以上的巨量外連紀錄，匯出引擎必須實作基於生成器 (Generator) 與分批載入 (`yield_per`) 的串流寫入機制。無論是單純的 CSV 匯出或完整的 ZIP 壓縮包，均須採行低記憶體佔用的邊讀邊寫模式，將資料直接串流至 HTTP 回應或磁碟中，防範 OOM 例外。
* **原生 SQL 聚合與前端渲染保護 (Native SQL Aggregation & UI Protection)**：處理一對多巢狀報表時，後端利用資料庫原生的 JSON 聚合與物件建構函數（如 SQLite 的 `json_group_array`/`json_object` 與 PostgreSQL 的 `json_agg`/`json_build_object`），將分組聚合運算下放給資料庫，以避免將巨量資料加載至 Python 記憶體中，並能恢復原生的 `LIMIT` 與 `OFFSET` 極速分頁。同時，為防前端 DOM 節點與傳輸過載，後端 API 會強制將子清單截斷至最多 10 筆，而前端 UI 預覽則進一步限制僅顯示前 n 筆（其餘以「及其它」標示）（目前設定 n 為 5），並提供繞過限制的 CSV/JSON 完整匯出 API，兼顧極速渲染與資料完整性。

### 10.2 資料庫併發與效能優化

* **資料庫批次操作 (Batch Operations) 嚴禁 N+1 查詢**：在處理大量連結資料（如外連去重比對、內連狀態寫入）時，嚴禁在迴圈內逐一執行單筆查詢 (N+1 Query)。必須強制採用「批次查詢 (Batch Query)」與「批次寫入 (Bulk Insert/Update)」，將資料庫 I/O 往返成本降至最低。若需防範寫入重複資料，應盡可能依賴資料庫層級的 Upsert (`INSERT ... ON CONFLICT DO NOTHING`)，並必須透過 `crawler/utils.py` 封裝跨資料庫 (SQLite 與 PostgreSQL) 的方言差異，保持業務層程式碼純淨。
* **PostgreSQL 連線池效能調校 (Connection Pool Optimization)**：系統在初始化資料庫引擎時，會動態偵測連線字串。若為 PostgreSQL，將自動啟用進階連線池參數（如 `pool_size`, `max_overflow`），並可透過 `.env` 環境變數進行調校。同時，強制啟用 `pool_pre_ping` 機制，在每次從連線池取出連線前，主動發送輕量級 `SELECT 1` 測試連線是否存活，若已中斷則自動重連，徹底解決因網路波動或資料庫重啟導致的服務中斷問題，確保生產環境下的高併發穩定性。
* **併發寫入優化**：針對頻繁的佇列狀態變更與結果寫入，資料庫連線需啟用高併發寫入優化，減少磁碟 I/O 頻率並避免寫入衝突。
  * **例外處理與事務回滾 (Exception Handling & Transaction Rollback)**：為應對高併發寫入時可能發生的各種衝突（如 SQLite 的 `database is locked` 或 PostgreSQL 的 Deadlock / 違反唯一約束等），系統必須：1. 針對 SQLite 提供可配置的鎖定等待逾時 (`timeout`) 以降低檔案鎖衝突機率。2. 無論使用何種資料庫，當捕捉到資料庫層級例外 (`SQLAlchemyError`) 時，必須強制執行事務回滾 (`rollback`)，防止 ORM Session 狀態失效導致後續操作引發級聯崩潰。
* **巨量資料刪除與 OOM 防禦 (Bulk Deletion & OOM Prevention)**：
  * **被動級聯刪除 (Passive Deletes)**：為防範刪除大型任務時，ORM 將數十萬筆的子紀錄（如佇列與外連結果）載入記憶體逐筆刪除而引發伺服器 OOM 崩潰，資料庫層級必須強制設定外鍵之 `ON DELETE CASCADE`，並於 ORM 關聯明確設定 `passive_deletes=True`，將級聯刪除操作全權交由底層資料庫引擎極速處理。
  * **無同步批次刪除 (Evaluate-Free Bulk Deletion)**：執行資料表批次刪除（如任務重置）時，ORM 之 `delete()` 操作必須強制搭配 `synchronize_session=False` 參數，略過極耗資源的記憶體物件比對與狀態同步，確保直接發送純 SQL 指令至資料庫，將 Python 記憶體消耗降至最低。
* **高併發複合索引優化 (Composite Indexing for High Concurrency)**：
  * **減少資料庫排序與全表掃描成本**：為解決在 1GB RAM 等微型雲端主機環境下 PostgreSQL 的 CPU 過載問題，系統對頻繁查詢的資料表強制建立「複合索引 (Composite Index)」。如在 `CrawlQueue` 建立 `(job_id, status, id)` 索引，消除爬蟲高頻獲取任務時的排序負擔；在 `ExternalLink` 建立 `(job_id, created_at)` 與 `(job_id, http_status_code, ip_address)` 索引，使巨量資料分頁與狀態統計能完全透過 Index Scan 執行，避免全表掃描引發 CPU 與記憶體風暴。此策略亦完全相容並加速 SQLite 的本地查詢。
* **統一狀態分類與預先計算 (Status Category Pre-computation)**：
  * 為消弭動態運算造成的效能瓶頸，並統一全系統對「連結狀態」的認知，`crawl_queue` 與 `external_links` 資料表均須實作 `status_category` 欄位。
  * 系統需在爬蟲寫入或更新紀錄時，根據 HTTP 狀態碼、IP 位址與錯誤訊息，預先將連結分類（如 `healthy`, `blocked`, `not_found`, `connection_error` 等）計算完成並實體化寫入資料庫中，並為其配置複合索引 `(job_id, status_category)`。
  * 未來無論是前台儀表板統計、API 分頁查詢或是 CLI 報表匯出，皆須直接依賴此預先計算之欄位，以 O(1) 複雜度或極高效率的索引掃描完成資料聚合與篩選。

### 10.3 應用層快取優化 (Application-Level Caching)

* **靜態報表快取 (Static Report Caching)**：針對已完成 (`completed`) 或異常終止 (`error`) 的任務，由於其報表結果已經靜止，系統應在後端 API 導入應用層記憶體快取（具備 TTL 機制），大幅降低重複切換分組模式與讀取儀表板時的 CPU 運算與資料庫 I/O，以達毫秒級別的極速回應。
* **高基數查詢防禦 (High Cardinality Defense)**：快取機制應精準鎖定「參數組合極少」且「CPU 運算密集」的聚合端點（如 `summary`、`diff`），嚴禁快取參數組合無限大的分頁清單與搜尋 API，以避免無效快取佔用記憶體（快取污染）甚至導致 OOM。
* **即時自動失效機制 (Instant Invalidation)**：當任務狀態變更（例如觸發重新探取 / Retry）或任務實體發生異動時（透過偵測資料庫 `updated_at` 欄位），對應的任務快取必須立即且自動失效，確保使用者永遠取得最新的探測結果。

## 11. 系統配置規範 (System Configuration)

### 11.1 配置層級、合併邏輯與安全防禦

* **全域與個別配置優先級**：系統提供全域配置定義預設值，並提供個別任務配置自訂參數。個別任務配置原則上優先權高於全域配置，但必須受到全域安全上下限之規範。
* **設定值聯集合併規則**：為了確保全域安全政策得以落實，對於以下設定項目，系統必須採用「全域與個別配置聯集合併 (Union)」的邏輯，而非直接覆寫：
  * **忽略副檔名 (`ignore_extensions`)** 與 **忽略正規運算式排除路徑 (`ignore_regexes`)**：任務配置與全域配置之清單自動聯集合併。
  * **自簽憑證豁免網域 (`ssl_exempt_domains`)**：任務配置、全域配置及環境變數之設定值自動聯集合併。
* **設定參數安全上下限強制限制 (Limit Enforcement)**：全域配置中可定義 `min_timeout`、`max_timeout`、`min_delay`、`max_delay`、`min_retries` 與 `max_retries` 等上下限安全閾值。若個別任務設定檔中的參數超出此安全閾值，系統在啟動任務時必須主動發出警告，並強制將其數值修正至符合全域安全範圍內以執行。
* **網域特定延遲之匹配原則**：網域特定延遲時間對照表 (`domain_delays`) 支援子網域匹配，且在比對時必須遵循「最長匹配優先原則」（例如，針對 `sub.example.com` 爬取時，若設定中同時存在 `example.com` 與 `sub.example.com` 的延遲設定，應優先套用較長之 `sub.example.com` 的延遲秒數）。
* **資安路徑防禦**：全域配置與任務配置的載入必須限制在指定的安全目錄內，系統需使用真實絕對路徑檢查，阻絕路徑遍歷（Path Traversal）或符號連結（Symlink）等資安漏洞。
* **防禦性型別轉換與設定檔容錯 (Defensive Type Coercion)**：由於全域設定與任務設定皆可能來自人為撰寫的 YAML 格式，系統在合併配置前，必須強制實施防禦性型別收斂。主動防堵如布林值混充數字、字串型態的數值、或非預期的資料結構。無法安全轉換之無效設定將被丟棄並降級至系統預設值，徹底防範因手動編輯配置錯誤導致爬蟲核心引發 `TypeError` 崩潰。
* **路徑智慧自動補齊**：為了提升使用易用性，當指定的任務設定檔路徑沒有完整前綴且為單純檔名時，系統必須能自動拼接至指定的安全目錄（如 `job/`），同時不破壞底層的真實絕對路徑安全檢驗。

### 11.2 日誌管理

* **防禦性日誌紀錄 (Defensive Logging)**：為了防禦帳號列舉，系統安全驗證（如密碼重設）即使帳號不存在也應維持相同行為。但在寫入系統日誌時，必須明確標記此類操作的真實語境（例如加註 `anonymous_attempt`），並確保背景清理任務（如 Zombie Job 清理）記錄觸發來源 (Caller)，避免維運人員或開發者在查閱日誌時缺乏上下文而造成誤判。
* **告警與關鍵字標籤策略 (Alert Tagging Strategy)**：當系統遭遇可能導致資料庫不一致、任務崩潰或日誌遺失的嚴重異常時，必須提升至 `logger.critical(..., exc_info=True)` 層級。此外，必須在日誌開頭強制加上特定的大寫結構化標籤（例如 `[DATA_INCONSISTENCY_ALERT]`, `[SCHEDULER_FAILURE]`, `[JOB_CRASH_ALERT]` 等），以利未來無縫整合 ELK、Datadog 等監控系統，快速建立精準的 PagerDuty 警報規則。
* **結構化任務統計日誌 (Structured Task Observability)**：為強化系統可觀測性與除錯能力，當爬蟲任務順利完成 (`completed`) 或異常中斷 (`error`) 時，系統必須在後端日誌中寫入結構化的統計摘要（包含：總爬取頁數、外部連結探測數量、重試次數與任務總耗時等）。嚴禁僅紀錄空泛的「任務完成」，確保維運人員能從日誌中建立效能基準 (Benchmark) 並迅速判別異常指標。
* **日誌輪轉與限制 (Log Rotation)**：系統日誌需同時輸出至主控台與日誌檔案，且必須實作日誌檔案的自動輪轉機制 (Log Rotation)，明確限制單一日誌檔案的大小上限（如 10MB）與歷史保留份數，防止長期運行佔滿磁碟空間。
* 日誌級別與路徑需支援動態配置。

### 11.3 CLI 操作與匯出導出規範

* **參數命名**：輸出報表或結果匯出的相關參數，強制使用完整長參數命名，不提供單一字母縮寫，以防操作混淆。
* **導出篩選器與統一過濾分類 (Unified Taxonomy)**：結果匯出必須支援依狀態進行篩選。為避免系統各模組認知歧異，系統必須訂定嚴格的狀態過濾字典（如區分 `dns_failed`, `connection_error`, `blocked`, `server_error` 等）。全系統上下（包含 CLI 選項、API 參數、Web UI 面板及資料庫底層 `status_category` 欄位）必須絕對遵從此統一之分類邏輯與命名對應關係。
* **去重聚合導出**：支援多維度的外連結果去重聚合導出，除原本的依目標頁面 (Target) 合併來源頁面外，需支援依來源頁面 (Source) 聚合列出其包含之所有外連，以及依目標網域 (Domain) 聚合統計其出現次數與獨立網址數。
* **排除網域過濾**：在匯出報表時，系統必須支援自訂排除目標網域清單的功能，以便過濾掉已知安全或不需關注的外部連結。
* **完整任務報表匯出**：系統必須支援單一指令匯出完整的任務報表壓縮檔 (ZIP)，其中需包含完整的網頁爬取巡覽紀錄 (`crawl_records.csv`)、外部連結清單 (`external_links.csv`)，以及內部失效連結診斷清單（含失效樣態分類），以利離線稽核與歸檔。
* **多格式支援**：支援將結果以 JSON 格式輸出至標準輸出，或直接匯出為 JSON / CSV 檔案。
* **支援 JSON 輸出 (`--json`)**：除結果匯出外，CLI 的狀態報告 (`--report`) 與任務清單 (`--list-jobs`) 應支援 `--json` 參數，以便與外部自動化腳本無縫整合。至於任務生命週期操作（如暫停、刪除、重置等），則依循標準 CLI 慣例，透過 Exit Code 與標準輸出日誌進行狀態反饋。
* **溫和暫停機制 (`Ctrl+C`)**：CLI 執行期間攔截 `KeyboardInterrupt` (Ctrl+C)，並轉化為溫和暫停指令，確保當前處理狀態安全寫入後再退出。
* **伺服器單一啟動入口 (`--serve`)**：CLI 需提供 `--serve` 選項作為統一的伺服器啟動入口，簡化整體服務啟動流程。

## 12. 系統維運與資料可攜性 (System Ops & Portability)

* **跨資料庫與跨環境任務遷移 (Job Sync)**：系統支援將單一任務的元資料、佇列與外連結果以 JSONL (JSON Lines) 格式無損打包為 ZIP 壓縮檔。該備份檔具備高度相容性，支援**跨資料庫（如 SQLite ↔ PostgreSQL）匯出與匯入**，並於匯入時自動配發全新的 `job_id` 與動態指定接手之使用者，完美解決跨機房轉移與資料交接的痛點。
  * **任務資料遷移便利包 (`job_sync.sh`)**：系統除支援直接呼叫 python 腳本外，必須配備 `scripts/job_sync.sh` shell 腳本便利包。該工具需支援自動進入虛擬環境（`.venv`），並提供清晰的 `export` 與 `import` 指令調度，以利運維人員在無後端環境下快速完成任務資料的備份與接手。
* **資料庫平滑升級支援 (Database Migration)**：內建 `migrate_sqlite_to_pg.py` 全自動移轉腳本。支援一鍵將開發環境的 SQLite 資料（含使用者、Session、歷史任務與巨量爬取結果）無痛遷移至生產環境的 PostgreSQL，並自動更新所有關聯與 Sequence 序列值，確保高併發環境的無縫升級。

## 13. CI/CD 與開發規範 (CI/CD & Development Standards)

* **自動化整合測試 (Pytest) 與模組級隔離**：專案必須採用一鍵式自動化整合測試套件。為防測試污染與殘留，必須實作以 Fixture 為核心的「模組級隔離架構」。在每個測試模組執行前，強制重設所有後端全域單例物件（Singletons，如 DB Engine）、清除 FastAPI 的依賴覆寫，並刷新環境變數快取，確保每個測試檔案皆在絕對乾淨的沙箱環境中運行。
* **基於 Playwright 的前端 E2E 測試架構與無頭模擬 (Playwright E2E Architecture & API Mocking)**：前端必須建構完善的全鏈路 E2E 測試架構（基於 Playwright 搭配 pytest-asyncio），涵蓋登入驗證、任務生命週期、表單操作與日誌管理等核心情境。為確保測試穩定性，在驗證介面互動與任務生命週期時，必須採用 API 攔截技術動態模擬後端狀態（如任務 `running` 與進度流轉），嚴禁在 UI 測試中真實觸發並等待後端爬蟲子程序執行。此舉能徹底斬斷前後端測試耦合，避免真實網路 I/O 造成測試結果不穩定 (Flaky Tests)。
* **靜態分析、排版檢驗與靜態型別檢查 (Static Analysis & Type Checking)**：提交之程式碼須於 CI/CD 流程中通過嚴格的靜態分析。包含使用 Ruff 進行排版與規則檢查、Pylint 進行深度的程式碼品質與壞味道分析，以及必須整合 **Mypy 靜態型別檢查** 確保所有函式與變數皆符合型別提示 (Type Hints) 規範，以大幅提升程式碼的健壯性與長期維護性。
* **自動化狀態反饋**：測試套件在完成檢驗後，必須能依照校驗結果（如 Queue 數量統計、DNS 異常捕獲、HTTP 狀態碼及去重聚合結果）自動回傳 Exit Code (0 代表成功，非 0 代表失敗)，以利 CI/CD 系統判定測試狀態。
* **依賴與環境鎖定**：所有的依賴軟體套件必須明確固定版本號，以確保在 CI/CD 容器中執行時的環境重現性與穩定性。
* **精準例外捕抓與錯誤隔離 (Precise Exception Handling)**：系統核心模組（包含 CLI、爬蟲引擎與任務管理器）嚴禁使用全域無差別捕捉例外（如 Pylint 警告之 `broad-exception-caught`）。各模組在實作錯誤處理時，必須精確標示捕捉的例外型別（例如 `yaml.YAMLError`、`OSError`、`SQLAlchemyError` 等），以確保能精確隔離錯誤邊界，保障系統異常崩潰追蹤與容錯處理的正確性。
* **API 規格同步自動化 (Automated OpenAPI Spec Generation)**：專案配置有 `scripts/gen_api_doc.py` 自動化工具，可直接從 FastAPI 實例提取最新的 Schema 並匯出為完整的 `openapi.json` 與 `api_spec.md`。為徹底防範前後端規格脫鉤，開發期間若有增修 API 路由與 Schema，皆必須執行此腳本以確保文件與程式碼維持絕對同步。

## 14. 本機測試網站與測試案例規格 (Test Sandbox Specifications)

* **測試伺服器**：
  * 為在開發與維護期間進行完整的 End-to-End 功能驗證，本專案需配備模擬測試網頁與 Mock 響應之本機測試伺服器。
  * 該伺服器需能模擬各種複雜的 HTTP 回應，以完整涵蓋與驗證爬蟲在重導向基準路徑修正、超時處理、智慧型重試、副檔名與媒體內容過濾、以及 User-Agent 偽裝等核心需求。
* **自動化測試啟動腳本**：
  * 專案需配備一鍵式自動化測試腳本，用以簡化開發測試與支援 CI/CD 程序。
  * 腳本需在背景啟動本機測試伺服器、清除歷史測試資料、執行爬行任務，並在結束後自動執行結果斷言校驗（檢查去重聚合數量、狀態碼捕獲與 DNS 異常紀錄等）。
  * 測試案例（如 `test/run_test.py`）必須涵蓋後端 (backend) 所有的 API 端點 (Endpoints)，確保各項 API 功能與邊界條件皆經過充分驗證。
  * 無論測試成功與否，皆須在結束前確保本機測試伺服器被安全關閉，並依照測試結果返回對應的 Exit Code (0 代表成功，非 0 代表失敗)，以利與自動化集成流程整合。
* **單一 URL 診斷除錯工具**：
  * 開發與維護環境中需配備獨立的單一 URL 測試腳本 (`scripts/test_url.py`），允許指定單一目標網址，以偵錯模式（Debug Mode）直接抓取並印出其內、外部連結解析與連線回應診斷結果，以利快速排查特定網頁的爬取行為。此工具亦應於 VS Code 開發環境配置對應的除錯啟動項目 (`.vscode/launch.json`），支援動態帶入自訂參數。

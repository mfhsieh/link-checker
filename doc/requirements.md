# 外部連結檢查爬蟲 - 系統需求規格書 (Requirements)

本文件詳列「外部連結檢查爬蟲」的各項核心需求、系統架構、容錯機制及開發規範。本文件將隨著專案演進持續更新，作為未來擴充功能與架構修改的最高指導原則。

---

## 1. 專案目的
建立一套高可靠性的網站爬蟲系統，用於自動巡覽特定網站，並找出連向外部（非指定網域）的連結，最終解析並紀錄該外部連結對應的 IP 位址。

---

## 2. 系統基本與資安需求 (Basic & Security Requirements)

* **資安防護規範**：
  * **防禦計時攻擊 (Timing Attack)**：系統在處理任何敏感憑證的驗證（如登入密碼驗證、CSRF Token 比對）時，必須確保執行時間具備恆定性。字串比對須強制使用安全的時間恆定比較函式（如 `secrets.compare_digest`）；密碼驗證時，無論帳號是否存在，皆須執行等效耗時的雜湊運算，防範攻擊者透過測量回應時間探測帳號狀態。
  * **SSRF 防禦與底層解析攔截**：為了防止伺服器端請求偽造 (SSRF) 攻擊，在發送 HTTP 請求時必須確保底層 Socket 連線使用的是已驗證安全的 IP 位址。禁止僅透過字串替換網址或變更 Host 標頭的方式繞過檢查，應實作 Socket 層級的 DNS 解析攔截或覆寫 (Monkey Patch) 機制以確保絕對安全。
  * **路徑防禦與輸入安全**：系統載入任何設定檔或處理外部路徑輸入時，必須強制進行安全路徑檢驗（如使用絕對真實路徑比對），以防制路徑遍歷（Path Traversal）或利用符號連結（Symlink）繞過檢驗所造成的敏感檔案外洩漏洞。
  * **安全審查與網域白名單**：系統應具備外部連結的安全稽核需求，支援網域白名單檢驗機制，以利發現並篩選出不在信任白名單內的超連結，防範潛在供應鏈資安風險。
  * **傳輸安全與憑證檢驗**：系統需具備識別並標記明文傳輸協定（如 HTTP 而非 HTTPS）之能力，以利傳輸安全性稽核；系統預設應對連線進行嚴格的 SSL/TLS 憑證鏈校驗，同時需保留針對特定信任之自簽憑證網域的配置豁免通道。
  * **機密性與配置憑證保護**：系統日誌與異常輸出中嚴格禁止明文紀錄或印出任何敏感憑證（如代理伺服器帳密）；載入此類機密配置時，應優先支援自環境變數（Environment Variables）讀取。
  * **匯出安全與注入防禦 (Export Security & Injection Defense)**：系統在匯出 CSV 報表時，必須強制對所有字串欄位進行跳脫處理 (Sanitization)，防止因惡意網址或內容以 `=`, `+`, `-`, `@` 等字元開頭，導致試算表軟體（如 Excel）將其作為公式執行的 CSV 注入攻擊 (CSV Injection / Formula Injection)。
  
* **CI/CD 自動化與測試規範**：
  * **自動化整合測試**：專案必須隨附一鍵式自動化整合測試套件，用以在 CI/CD 自動化整合管道中進行爬蟲系統的 End-to-End 功能驗證與容錯校驗。
  * **自動化狀態反饋**：測試套件在完成檢驗後，必須能依照校驗結果（如 Queue 數量統計、DNS 異常捕獲、HTTP 狀態碼及去重聚合結果）自動回傳 Exit Code (0 代表成功，非 0 代表失敗)，以利 CI/CD 系統判定測試狀態。
  * **依賴與環境鎖定**：所有的依賴軟體套件必須明確固定版本號，以確保在 CI/CD 容器中執行時的環境重現性與穩定性。

## 3. 爬蟲核心功能需求 (Crawler Core Functional Requirements)

* **指定起點 (Start URL)**：系統需支援設定單一網址作為網頁爬取的起始起點。
* **廣度優先巡覽 (Breadth-First Search)**：系統在探索網站內部網頁時，必須強制採用**廣度優先搜尋 (BFS)** 演算法進行巡覽，以確保巡覽順序的預測性與深度的精確控制：
  * **逐層探索機制**：爬蟲必須以起始網址為第 0 層，逐層向外探索。在完整處理並爬取完當前深度層級的所有待爬網址之前，不得發起下一深度層級的網頁請求。
  * 佇列順序管理：所有新探索到的內部連結應依序加入待爬網址佇列的末端，並嚴格遵循先進先出 (FIFO) 的順序進行處理。在讀取資料庫佇列的待爬取項目時，必須強制按佇列識別碼 (ID) 進行升冪排序，以從資料庫層面保障 FIFO 的精確性。
  * **深度精確限制**：系統必須依據此 BFS 巡覽邏輯，精確計算每個網頁的探索深度。當網址深度達到配置的「最大爬取深度」限制時，系統僅能解析該網頁所包含的外部連結以完成該層的檢查，不得將該網頁內的任何內部網址加入待爬佇列，以確保深度限制與資源控制能精確且有效地執行。
* **網域內遍歷限制**：爬蟲需限制在「指定之特定網域」內進行網頁巡覽，防止無限制地爬取非指定網域的頁面。
* **重新導向 (Redirect) 處理**：當遇到 HTTP 重新導向時，爬蟲需追蹤跟隨至最終目標網址，並以該「最終實際網址」作為基準網址（Base URL）解析 HTML 中的相對路徑，避免拼湊出無效的連結。
* **外部連結與資源資產偵測與紀錄**：
  * **超連結與外連資源掃描**：系統除掃描網頁中的外部超連結（`<a>` 標籤的 `href`）外，為達成防範**失效連結劫持（Broken Link Hijacking）**等供應鏈資安防護目的，亦必須主動掃描網頁中所參照之各式外連靜態與動態資源。
  * **支援掃描之標籤與屬性**：
    * 載入資源類（`src`）：`<script>`、`<iframe>`、`<img>`、`<embed>` 等標籤之外連資源。
    * 樣式表與關聯類（`href`）：`<link>` 標籤之資源。
    * 表單提交類（`action`）：`<form>` 標籤指向之外連目的地，用以防範表單敏感個資外洩至已失效的外連網域。
    * 物件類（`data`）：`<object>` 標籤所嵌入之外部檔案。
  * **欄位紀錄**：詳細記錄其目標網址、來源網址，並解析該目標外連網址主機之 IP 位址，同時進行 HTTP 存活檢查與白名單稽核。
* **內容過濾與資源節約**：
  * **大檔案攔截與記憶體保護**：爬蟲下載網頁內容時，為防止伺服器回傳無限資料流導致記憶體耗盡 (OOM)，必須強制採用分塊讀取串流 (Chunked Stream) 模式，且須設定「最大下載容量上限」(如 10MB)。一旦資料超過上限，必須立即截斷並只保留已下載的內容。
  * **靜態副檔名過濾**：支援自訂的副檔名忽略規則（如 PDF、ZIP 等），避免對非 HTML 網頁資源發送不必要的請求。
  * **規則排除**：支援利用規則運算式（Regex）排除特定的網址路徑，直接忽略符合規則的連結。
  * **媒體內容攔截**：發送請求後需優先檢查回應的內容類型（Content-Type）。若目標為非 HTML 網頁媒體資源，必須立即中斷連線，停止下載檔案內容以節約頻寬。
* **HTTP 狀態與請求延遲機制**：
  * 系統需精準追蹤並記錄每次網址爬取的 HTTP 狀態碼與連線錯誤。
  * **區分跳過情境與延遲等待**：
    * 針對因「副檔名」或「排除路徑」而被跳過的網址（未實際發出 HTTP 請求），系統應標記為跳過且不進行延遲等待，立即處理下一筆以提升效率。
    * 針對因「媒體內容不符」而在讀取標頭後中斷連線的網址（已實際發出 HTTP 請求），系統應標記為跳過並套用延遲等待，避免對目標伺服器造成連線壓力。
* **網址與外連去重機制**：
  * 巡覽佇列需具備防重機制，確保同一個任務內不重複爬取相同的內部網頁。
  * 外部連結紀錄需具備去重機制，針對相同的「來源網址 -> 目標外連網址」不重複寫入。系統應在 Crawler DB 的 `external_links` 資料表層級對 `(job_id, source_url, target_url)` 加上實體 UniqueConstraint 約束，且在插入資料庫前於程式中預先進行去重查重，防止在高並發或重試機制下寫入衝突。
* **網域 IP 解析優化**：
  * 解析外連 IP 時，系統需擷取目標主機網域進行 DNS 查詢，而非針對完整網址，以減少 DNS 伺服器查詢負擔。
* **瀏覽器偽裝**：
  * 系統需支援自訂 User-Agent 標頭，預設應偽裝為主流網頁瀏覽器，防止被目標網站因預設連線標頭而阻擋。
* **安全傳輸協定稽核 (is_secure)**：
  * **主動識別明文協定**：系統在探索並紀錄外部超連結與靜態資源時，必須主動分析其傳輸協定（Scheme）是否為安全傳輸協定（如 HTTPS）。
  * **持久化與稽核匯出**：非安全傳輸協定（如明文 HTTP）必須被明確標記並儲存於資料庫中，且必須作為結果匯出（CSV/JSON）時的必要檢視欄位，以利資安管理員快速稽核傳輸安全。
* **智慧型錯誤重試與指數退避**：
  * **錯誤分類重試**：遇到永久性錯誤（如 404、403 等）時直接標記為失敗，不進行任何重試；僅對暫時性連線異常或暫時性伺服器錯誤（如 503、連線逾時）進行重試。
  * **指數退避機制**：重試時應引入退避等待機制，隨重試次數增加等待時間，避免連續請求加重目標伺服器負載。
* **外部連結 HTTP 存活檢查與診斷**：
  * 針對探索到的所有外部連結，系統需進行輕量級的 HTTP 存活探測，並記錄其 HTTP 狀態碼與連線錯誤。
  * **雙層探測與繞過機制**：優先發送 HEAD 請求。若目標網址回傳特定的阻擋狀態碼 (如 400, 403, 405) 或是屬於已知設有反爬蟲限制的大型平台 (如社群媒體等)，系統必須自動降級使用帶有微量傳輸範圍限制 (如僅讀取前 1024 位元組的 Range 標頭) 的 GET 串流請求，以模擬正常瀏覽器特徵並繞過反爬蟲 WAF 機制，減少錯誤的損毀連結判定。
  * **任務級存活檢查快取機制**：為了防止對相同的外部網址重複發起大量 HTTP 連線探測 (如在網站中被重複引用數百次)，系統必須實作任務級的健康檢查快取。當同一個外部連結在當次任務中已被探測過，後續引用應直接複用其結果，以節省系統與外部目標網站的頻寬，降低被封鎖風險。
  * **多工並發檢測機制**：系統在探測單一網頁解析出的外部連結清單時，必須強制採用並發檢測技術，以防止同步依序探測所造成的執行緒阻塞與爬行遲滯，提升檢測整體效能。
* **爬行禮貌與頻率控制**：
  * 系統必須實作可調控的網頁請求延遲機制，並支援對不同主機網域進行頻率限制，防止因過度存取對內部或目標網站伺服器造成非預期的負載壓力與服務中斷。
* **資源限制與死循環陷阱防禦 (Resource Limitation & Crawl Trap Defense)**：
  * 系統必須能針對單一任務配置「最大爬取深度」與「最大抓取頁面數量上限」之硬性限制，在超出閾值時必須主動優雅終止任務，防止因網站結構陷阱（Crawl Trap）導致系統資源耗盡。

## 4. 任務管理與容錯機制 (Job Management & Fault Tolerance)

爬蟲系統面臨長時間運作，必須具備狀態驅動的容錯能力與中斷接續能力。

* **狀態驅動設計**：
  * **子程序管理與狀態持久化**：爬蟲任務的背景執行子程序狀態不應依賴伺服器記憶體變數，應透過寫入實體 PID 檔案機制實作持久化追蹤。系統需能透過 PID 精確判斷子程序是否存活，並於伺服器重啟或異常中斷後能安全地接管或清理。
  * 任務層級狀態需區分掛起、執行中、暫停、完成與異常。
  * 網址佇列層級狀態需區分等待中、已完成、失敗與跳過。
* **中斷與恢復 (Pause & Resume)**：
  * 系統需能應對執行中的非預期中斷，並提供透過任務識別碼恢復未完成佇列繼續執行的機制。
* **設定檔持久化**：
  * 任務建立時，系統需將該次任務合併之最終執行配置快照儲存至資料庫中。未來恢復執行時，強制讀取資料庫內的配置快照，確保參數一致性。
* **執行狀態安全鎖與強制接管**：
  * 為防止多程序同時恢復同一個任務造成數據競爭或資料庫鎖死，系統在啟動時需進行狀態防呆檢查，拒絕執行狀態為「執行中」的任務。
  * 針對意外卡死的任務，系統需提供強制接管的安全解鎖開關。
* **資料庫併發優化**：
  * 針對頻繁的佇列狀態變更與結果寫入，資料庫連線需啟用高併發寫入優化，減少磁碟 I/O 頻率並避免寫入衝突。
* **任務生命週期管理與多使用者隔離**：
  * **多使用者任務隔離**：系統支援在建立或恢復任務時，將任務綁定至特定的使用者識別碼 (`user_id`)，以利多使用者共享系統時的任務歸屬辨識，且支援依使用者識別碼對所有任務進行列表與查詢過濾。
  * **暫停任務**：支援溫和的暫停信號，在完成當下網址爬取後安全暫停。
  * **刪除任務**：支援清理指定任務的所有佇列與結果，防止資料無限制膨脹。
  * **重置任務**：支援將已完成或失敗的任務恢復至初始狀態，以利重新爬取。

## 5. 系統架構與解耦原則 (System Architecture & Decoupling Principles)

系統範圍涵蓋以下模組，且設計需遵循解耦原則：

1. **網站前台**：提供使用者登入、任務操作介面與檢視結果報告。
2. **網站後台**：提供管理員進行使用者管理（邀請、停用）、系統設定與任務監控。
3. **爬蟲程式核心**：專責網頁抓取、HTML 解析與網域 DNS 查詢。
4. **任務管理器**：負責任務排程、佇列狀態調度、資料持久化與異常處理。

### 5.1 資料庫分離架構

系統需維護兩個**實體分離**的資料庫，彼此不得共用連線池或混用 Schema：

| 資料庫 | 用途 | 存取方 |
|--------|------|--------|
| **帳號資料庫**（Auth DB） | 儲存使用者帳號、邀請憑證、Session、角色、登入日誌等身分驗證相關資料 | 前台、後台 Web 層 |
| **爬蟲資料庫**（Crawler DB） | 儲存任務配置、網址佇列、爬取結果、外連紀錄等爬蟲業務資料 | 前台、後台 Web 層、CLI |

* 兩個資料庫的連線設定（路徑 / DSN）需分別獨立配置，透過環境變數或配置檔案各自指定。
* **關聯完整性保護**：針對預設未啟用外鍵檢查的 SQLite，系統建立連線初始化時，必須明確執行 `PRAGMA foreign_keys=ON;`，以確保外鍵約束強制生效。
* **跨庫最終一致性與軟刪除機制**：跨資料庫（Auth DB 與 Crawler DB）資源刪除時，為避免分散式事務失敗導致不一致，系統應採「軟刪除 (Soft Delete)」機制，於主庫標記刪除，再由定時任務 (Cronjob) 進行髒資料清理，確保最終一致性。
* 爬蟲核心與任務管理模組**僅能存取爬蟲資料庫**，不得直接讀寫帳號資料庫；身分驗證邏輯須完全隔離於 Web 層。
* 兩個資料庫的 Schema 遷移腳本需分別管理，可獨立執行。

> **⚠️ 核心獨立運作保證 (CLI-First)**：
> 爬蟲核心與任務管理模組必須具備高度獨立性。在無前後台網頁系統的環境下，依然能單獨透過命令列介面 (CLI) 啟動、維護與匯出資料。CLI 模式下僅需爬蟲資料庫，不依賴帳號資料庫。

## 6. 系統配置與開發環境規範 (System Configuration & Development Environment Specs)

* **配置層級、合併邏輯與安全防禦**：
  * **全域與個別配置優先級**：系統提供全域配置定義預設值，並提供個別任務配置自訂參數。個別任務配置原則上優先權高於全域配置，但必須受到全域安全上下限之規範。
  * **設定值聯集合併規則**：為了確保全域安全政策得以落實，對於以下設定項目，系統必須採用「全域與個別配置聯集合併 (Union)」的邏輯，而非直接覆寫：
    * **忽略副檔名 (`ignore_extensions`)** 與 **忽略正規運算式排除路徑 (`ignore_regexes`)**：任務配置與全域配置之清單自動聯集合併。
    * **自簽憑證豁免網域 (`ssl_exempt_domains`)**：任務配置、全域配置及環境變數之設定值自動聯集合併。
  * **設定參數安全上下限強制限制 (Limit Enforcement)**：全域配置中可定義 `min_timeout`、`max_timeout`、`min_delay`、`max_delay`、`min_retries` 與 `max_retries` 等上下限安全閾值。若個別任務設定檔中的參數超出此安全閾值，系統在啟動任務時必須主動發出警告，並強制將其數值修正至符合全域安全範圍內以執行。
  * **網域特定延遲之匹配原則**：網域特定延遲時間對照表 (`domain_delays`) 支援子網域匹配，且在比對時必須遵循「最長匹配優先原則」（例如，針對 `sub.example.com` 爬取時，若設定中同時存在 `example.com` 與 `sub.example.com` 的延遲設定，應優先套用較長之 `sub.example.com` 的延遲秒數）。
  * **資安路徑防禦**：全域配置與任務配置的載入必須限制在指定的安全目錄內，系統需使用真實絕對路徑檢查，阻絕路徑遍歷（Path Traversal）或符號連結（Symlink）等資安漏洞。
  * **路徑智慧自動補齊**：為了提升使用易用性，當指定的任務設定檔路徑沒有完整前綴且為單純檔名時，系統必須能自動拼接至指定的安全目錄（如 `job/`），同時不破壞底層的真實絕對路徑安全檢驗。
* **日誌管理**：
  * **日誌輪轉與限制 (Log Rotation)**：系統日誌需同時輸出至主控台與日誌檔案，且必須實作日誌檔案的自動輪轉機制 (Log Rotation)，明確限制單一日誌檔案的大小上限（如 10MB）與歷史保留份數，防止長期運行佔滿磁碟空間。
  * 日誌級別與路徑需支援動態配置。
* **CLI 操作與匯出導出規範**：
  * **參數命名**：輸出報表或結果匯出的相關參數，強制使用完整長參數命名，不提供單一字母縮寫，以防操作混淆。
  * **導出篩選器**：結果匯出必須支援依狀態進行篩選，包含篩選 DNS 失敗連結、HTTP 異常與連線失敗的損毀連結。
  * **去重聚合導出**：支援多維度的外連結果去重聚合導出，除原本的依外連目標 (Target) 合併來源頁面外，需支援依來源頁面 (Source) 聚合列出其包含之所有外連，以及依目標網域 (Domain) 聚合統計其出現次數與獨立網址數。
  * **排除網域過濾**：在匯出報表時，系統必須支援自訂排除目標網域清單的功能，以便過濾掉已知安全或不需關注的外部連結。
  * **完整任務報表匯出**：系統必須支援單一指令匯出完整的任務報表壓縮檔 (ZIP)，其中需包含完整的網頁爬取巡覽紀錄 (`crawl_records.csv`) 以及外部連結清單 (`external_links.csv`)，以利離線稽核與歸檔。
  * **多格式支援**：支援將結果以 JSON 格式輸出至標準輸出，或直接匯出為 JSON / CSV 檔案。
  * **全面支援 JSON 輸出 (`--json`)**：除結果匯出外，CLI 執行過程、狀態報告與所有操作回饋皆應支援 `--json` 參數，以便與外部自動化腳本無縫整合。
  * **溫和暫停機制 (`Ctrl+C`)**：CLI 執行期間攔截 `KeyboardInterrupt` (Ctrl+C)，並轉化為溫和暫停指令，確保當前處理狀態安全寫入後再退出。
  * **伺服器單一啟動入口 (`--serve`)**：CLI 需提供 `--serve` 選項作為統一的伺服器啟動入口，簡化整體服務啟動流程。
* **依賴鎖定**：
  * 專案的軟體套件依賴清單必須明確鎖定版本號，確保部署環境的穩定性與可重現性。

## 7. 本機測試網站與測試案例規格 (Test Sandbox Specifications)

* **測試伺服器**：
  * 為在開發與維護期間進行完整的 End-to-End 功能驗證，本專案需配備模擬測試網頁與 Mock 響應之本機測試伺服器。
  * 該伺服器需能模擬各種複雜的 HTTP 回應，以完整涵蓋與驗證爬蟲在重導向基準路徑修正、超時處理、智慧型重試、副檔名與媒體內容過濾、以及 User-Agent 偽裝等核心需求。
* **自動化測試啟動腳本**：
  * 專案需配備一鍵式自動化測試腳本，用以簡化開發測試與支援 CI/CD 程序。
  * 腳本需在背景啟動本機測試伺服器、清除歷史測試資料、執行爬行任務，並在結束後自動執行結果斷言校驗（檢查去重聚合數量、狀態碼捕獲與 DNS 異常紀錄等）。
  * 無論測試成功與否，皆須在結束前確保本機測試伺服器被安全關閉，並依照測試結果返回對應的 Exit Code (0 代表成功，非 0 代表失敗)，以利與自動化集成流程整合。

## 8. 使用者帳號管理與邀請制登入機制 (User Account & Invitation-Based Authentication)

本系統採邀請制（Closed Registration），禁止公開自由註冊，所有使用者帳號須由後台管理員建立並發送邀請。

### 8.1 邀請流程

* **後台建立帳號**：管理員於後台輸入受邀者的電子郵件信箱，系統自動產生一組唯一的 UUID 作為一次性邀請憑證，並將帳號狀態設定為「待首次登入（Pending）」。
* **邀請信件寄送**：系統自動組成邀請郵件，內含：
  * 受邀者的使用者帳號（電子郵件信箱）
  * 一次性邀請 UUID（用於身分驗證）
  * 系統登入網址（不包含 Token，防止瀏覽器歷史紀錄或 Referer 洩漏）
  * 邀請碼有效期限（可配置，如 72 小時）
* **邀請憑證安全性**：
  * UUID 須為單次使用；首次登入完成並設定密碼後，邀請 UUID 立即失效，不得再次使用。
  * 超過有效期限的邀請連結應自動失效，管理員可重新寄出新的邀請。

### 8.2 首次登入流程

1. **輸入邀請資訊**：使用者進入系統登入網址後，需手動輸入電子郵件與專屬邀請碼 (UUID) 進行驗證。
2. **身分驗證**：後端驗證電子郵件與 UUID 組合的有效性（帳號存在、狀態為 Pending、UUID 未失效）。
3. **強制設定密碼**：身分驗證通過後，系統立即導向「密碼設定頁面」，使用者**不得跳過**此步驟，在設定符合安全標準的密碼之前，無法進入任何功能頁面。
4. **密碼安全標準**：
   * 長度：最少 12 個字元。
   * 複雜度：需同時包含大寫字母、小寫字母、數字、特殊符號中的至少三項。
   * 禁用規則：禁止使用與帳號（電子郵件本地端）相同或過於相似的字串作為密碼。
   * 儲存：密碼需以強雜湊演算法（如 bcrypt / Argon2）加密後儲存，嚴禁明文或弱雜湊（如 MD5、SHA-1）。
5. **完成啟用**：密碼設定完成後，帳號狀態更新為「正常（Active）」，邀請 UUID 立即清除失效，系統自動導向主功能頁面。

### 8.3 日常登入流程

* **一般登入**：已啟用帳號使用電子郵件 + 密碼登入。
* **Session 安全**：
  * 登入成功後簽發具有效期限的 Session Token（或 JWT）。
  * 前端未經身分驗證的請求一律重新導向至登入頁面。
* **登入失敗保護**：連續登入失敗達閾值（如 5 次）後，帳號暫時鎖定（如 15 分鐘），防範暴力破解。
* **登出**：提供明確的登出機制，清除 Session / Token。

### 8.4 密碼變更

* 已登入的使用者可主動在「帳號設定」中修改密碼，需先輸入現有密碼進行驗證後才可設定新密碼，新密碼同樣需符合 8.2 節的密碼安全標準。

### 8.5 帳號狀態管理

| 狀態 | 說明 |
|------|------|
| Pending | 已建立帳號、邀請信已寄出，尚未完成首次登入設密 |
| Active | 帳號正常可用 |
| Suspended | 由管理員停用，禁止登入 |
| Expired | 邀請連結過期且尚未首次登入，需重新寄送邀請 |

### 8.6 帳號資料庫（Auth DB）Schema 規範

帳號相關資料**獨立存放於專屬的帳號資料庫**，與爬蟲資料庫實體分離（詳見第 5.1 節）。Auth DB 需涵蓋以下資料表：

* **`users`**：儲存使用者基本資料（id、email、password_hash、role、status、created_at、last_login_at）。
* **`invitations`**：儲存邀請憑證（id、user_id、uuid、expires_at、used_at、created_at）；UUID 使用後立即標記 `used_at`。
* **`sessions`**：儲存有效的 Session Token（token_hash、user_id、expires_at、created_at、ip_address、user_agent）；登出後立即刪除或標記失效。
* **`auth_logs`**：儲存登入 / 登出 / 失敗 / 鎖定等身分驗證事件，以及後台敏感的高風險管理操作事件（如全域配置變更、任務強制操作、使用者帳號狀態變更與刪除等）（user_id、event_type、ip_address、detail、created_at）。其 `detail` 欄位以 JSON 格式儲存詳細的操作前後差異與目標資訊。

> Session Token 本身不得以明文儲存於資料庫，需儲存其雜湊值（Hash），以防資料庫外洩時 Token 被直接複用。

## 9. 前台功能需求 (Frontend Functional Requirements)

前台面向一般受邀使用者，提供任務操作、結果查閱等核心功能。

### 9.1 身分驗證與帳號

* **登入頁面**：支援以電子郵件 + 密碼登入；首次登入可接受以電子郵件 + 邀請 UUID 登入。
* **首次登入密碼設定頁面**：如第 8.2 節所述，強制於功能頁面前完成密碼設定。
* **帳號設定頁面**：使用者可修改個人密碼（需先驗證現有密碼）。
* **登出功能**：提供登出按鈕，清除 Session 並返回登入頁面。

### 9.2 任務管理（前台）

* **任務列表頁面**：
  * 顯示當前使用者的所有任務，含任務名稱、起始 URL、建立時間、狀態（等待中、執行中、暫停、完成、異常）及進度摘要。
  * 支援依狀態篩選、依建立時間排序。
* **新增任務**：
  * 表單輸入起始 URL、最大深度、最大頁面數、延遲設定等參數。
  * 前端即時驗證 URL 格式合法性。
  * 系統需動態向後端請求全域預設配置與安全上下限，並自動填入表單作為預設值。
* **任務詳情頁面**：
  * 提供獨立的檢視對話框，顯示任務建立當時被鎖定的「執行配置快照（格式化排版）」。
  * 顯示執行進度：已爬頁面數 / 佇列總數、目前深度層級、已發現外連數量。
  * 提供操作按鈕：暫停、恢復、重置、刪除。
* **即時狀態更新**：任務執行中，前台應透過輪詢（Polling）或 WebSocket 即時反映任務執行進度。

### 9.3 結果查閱與匯出

* **外連結果列表**：
  * 以表格顯示所有已探索的外部連結，欄位包含：目標 URL、來源頁面、IP 位址、HTTP 狀態碼、是否安全（HTTPS）、是否在白名單內、DNS 狀態。
  * 支援依狀態（正常 / DNS 失敗 / HTTP 異常 / 非白名單）篩選。
  * 支援依目標網域、來源頁面關鍵字搜尋。
  * 支援去重聚合視圖：支援多維度聚合模式 (Target/Source/Domain) 合併與統計外連數據。
* **結果匯出**：
  * 支援將篩選後的結果匯出為 CSV 或 JSON 格式下載。
  * 支援完整報表 (ZIP 壓縮檔) 下載功能。
  * 支援於匯出時自訂排除指定的目標網域。
  * 匯出內容需與 CLI 匯出功能保持一致（欄位對齊）。
* **任務報告摘要**：提供任務維度的統計摘要，包含總爬取頁數、外連總數、損毀連結數、非白名單數、HTTP 異常數等。

### 9.4 使用者介面體驗

* 全站操作需具備清晰的錯誤提示（如 URL 格式錯誤、無權限等）。
* 長時間操作（如任務執行中）需提供明確的載入指示器（Progress Indicator）。
* 介面設計需具備基本的響應式佈局（RWD），支援主流桌面瀏覽器。
* **對話框防呆規範**：嚴禁使用瀏覽器原生之 `alert`、`confirm`、`prompt`，所有破壞性操作（如刪除、停用、重置）必須使用自訂之 Modal 視窗進行二次確認，並依照危險等級套用警示色彩。

### 9.5 前台技術實作規範

前台介面採**輕量原生技術棧**，以降低供應鏈風險、減少打包複雜度與長期維護成本：

* **JavaScript**：
  * 強制使用 **Vanilla JS**，禁止引入 React、Vue、Angular、Svelte 等前端框架。
  * 模組化採用瀏覽器原生 **ES Modules（ESM）**（`<script type="module">`），允許使用 `import` / `export` 進行程式碼拆分，無需打包工具（Bundler）。
  * 禁止使用 jQuery 或其他 DOM 操作函式庫。
* **CSS**：
  * 使用 **Vanilla CSS**（含 CSS Custom Properties / CSS Variables），禁止引入 Tailwind CSS、Bootstrap 等 CSS 框架。
  * 允許使用 CSS 原生 `@import` 或 CSS Modules（若伺服器端支援）進行樣式分拆。
* **第三方套件原則**：
  * 原則上**不引入第三方前端套件**；若確有必要（如圖表渲染），須經評估後於需求文件中明確列出，並鎖定版本與來源（使用 SRI Hash 驗證完整性）。
  * 禁止透過公共 CDN 動態載入未鎖定版本的第三方資源。
* **後端整合**：
  * 前台與後端透過 **REST API（JSON）** 溝通，前台不直接渲染伺服器端模板（Server-Side Template）。
  * API 請求統一透過封裝的 `fetch` 工具函式發送，集中處理身分驗證標頭、錯誤回應與 Token 刷新邏輯。

## 10. 後台功能需求 (Backend Admin Functional Requirements)

後台面向系統管理員，提供使用者管理、全域配置、系統監控等功能，存取需具備管理員角色（Admin Role）。

### 10.1 管理員身分驗證

* 管理員使用獨立帳號登入後台，或由角色權限區分（RBAC）控管存取。
* 後台路徑需與前台路徑明確隔離，未授權存取一律返回 403 或重導向至登入頁面。

### 10.2 使用者管理

* **使用者列表**：
  * 顯示所有帳號，含電子郵件、帳號狀態、建立時間、最後登入時間、角色。
  * 支援依帳號狀態篩選（Pending / Active / Suspended / Expired）。
* **新增使用者（邀請）**：
  * 輸入受邀者的電子郵件信箱。
  * 系統產生 UUID、設定有效期限，並寄出邀請信。
  * 可配置邀請有效期限（預設 72 小時）。
* **重新寄送邀請**：針對 Pending 或 Expired 狀態的帳號，產生新的 UUID 並重新寄出邀請信，舊 UUID 立即失效。
* **停用 / 啟用帳號**：管理員可切換帳號為 Suspended / Active 狀態，停用後該帳號立即無法登入。
* **刪除帳號**：管理員可永久刪除帳號及其所有關聯任務資料（需二次確認）。
* **角色管理**：支援將一般帳號提權為管理員（Admin），或將管理員降權為一般使用者（User）。

### 10.3 任務監控（後台）

* **全任務列表**：顯示全部使用者的所有任務（不限於當前管理員），可依使用者、狀態篩選。
* **任務強制操作**：管理員可對任意任務執行暫停、強制接管（解鎖卡死任務）、刪除操作。
* **執行狀態概覽**：提供全系統任務執行概覽，包含目前執行中任務數、佇列等待中任務數、異常任務數等。

### 10.4 全域配置管理

* 提供 Web 介面檢視與修改系統全域配置，包含：
  * 爬蟲安全上下限（Timeout、Delay、Retries）
  * 全域忽略副檔名清單（`ignore_extensions`）
  * 全域忽略路徑規則（`ignore_regexes`）
  * 自簽憑證豁免網域（`ssl_exempt_domains`）
* 配置修改需寫入安全操作日誌（`AuthLog`），詳細記錄修改的操作者 ID、時間、IP 位址以及修改前與修改後的配置內容差異（`detail` 欄位以 JSON 儲存）。

### 10.5 郵件配置管理

* 後台提供郵件伺服器（SMTP）設定介面（主機、埠號、帳號、密碼、寄件者名稱）。
* 支援「寄送測試郵件」功能，以驗證 SMTP 配置是否正常。
* SMTP 密碼於儲存時需加密，且在前端顯示時不得回傳明文。

### 10.6 系統日誌查閱

* 後台提供系統操作日誌查閱介面，涵蓋儲存於 `auth_logs` 的各類事件，包含：
  * 使用者登入 / 登出 / 鎖定等紀錄（含 IP、時間戳）
  * 帳號邀請與狀態變更紀錄（如停用、變更角色）與刪除帳號紀錄
  * 全域配置變更紀錄（記錄修改前後的詳細欄位差異）
  * 任務強制操作紀錄（如強制接管 `takeover`、強制刪除 `delete` 等）
* 日誌支援依時間範圍（API 需支援 `start_date` 與 `end_date` 參數以 ISO 8601 或 YYYY-MM-DD 格式進行精確的日誌時間區間過濾）、事件類型（`event_type`）與操作使用者（`user_id`）進行篩選。

### 10.7 後台技術實作規範

後台（Admin）介面與前台共用同一後端 API 服務，以角色權限（RBAC）控管存取，**不另建獨立後台應用**。後台前端頁面同樣遵循 §9.5 所訂之 Vanilla JS + ESM 技術規範：

* **後台路由隔離**：後台相關頁面路由統一以 `/admin/` 前綴區分，後端中介層（Middleware）強制驗證請求方是否具備 Admin 角色；缺乏權限時一律回傳 403，不洩漏任何後台資源。
* **操作確認保護**：高風險操作（刪除帳號、強制接管任務、重置配置）需在後台前端呈現二次確認對話框，並於後端 API 端再次驗證操作者角色，防止 CSRF 或誤觸。
* **配置頁面輸入驗證**：全域配置修改表單須於前端進行格式預驗（如正規運算式語法、數值範圍），並於後端 API 進行最終合法性校驗，兩者缺一不可。

## 11. 系統整體技術堆疊規範 (Overall Technology Stack)

本節定義整個系統（前台、後台、API、爬蟲核心）的技術選型原則，作為後續實作的最高指導依據。

### 11.1 後端（Python Web 服務）

* **語言**：Python 3.11+。
* **Web 框架**：採用輕量 Python Web 框架（如 Flask 或 FastAPI）；選型需確保能與現有爬蟲核心模組整合，且支援非同步 I/O（用於任務狀態推送）。
* **API 風格**：RESTful JSON API；所有端點需明確定義 HTTP 方法語義（GET / POST / PATCH / DELETE）。
* **身分驗證**：使用 HTTP-only Cookie 承載 Session Token（Token 值為 Auth DB 中 `sessions.token_hash` 的對應原始值）；禁止將 Token 儲存於 `localStorage` 或 `sessionStorage`。
* **CSRF 防禦**：對所有狀態變更類 API（POST / PATCH / DELETE）強制實施 CSRF Token 驗證（Double Submit Cookie 或 Synchronizer Token 模式）。
* **輸入驗證與過濾**：所有 API 端點的請求參數需於後端進行嚴格的型別與範圍驗證；接收設定參數時，必須採用**白名單 (Whitelist)** 機制過濾未知參數。
* **最小權限暴露 (Masking)**：後端在回傳系統設定或任務快照至前端時，必須主動過濾內部邏輯設定，並將敏感憑證（如 Proxy 密碼）進行遮蔽（Masking）處理。

### 11.2 資料庫

* **引擎選型**：
  * **Auth DB**：建議使用 SQLite（開發）或 PostgreSQL / MySQL（生產），以支援交易與行級鎖定。
  * **Crawler DB**：延續現有爬蟲資料庫引擎（SQLite），並遵循現有配置管理規範。
* **Schema 遷移**：使用遷移工具（如 Alembic）管理兩個資料庫的 Schema 版本，遷移腳本需納入版本控制並可在 CI/CD 中自動執行。
* **ORM / 查詢**：可選用輕量 ORM（如 SQLAlchemy Core）或直接使用參數化 SQL 查詢；**嚴禁字串拼接 SQL**，以防 SQL Injection。

### 11.3 前台技術堆疊（摘要）

詳見 §9.5，要點如下：

* Vanilla JS + 瀏覽器原生 ESM（無 Bundler）。
* Vanilla CSS + CSS Custom Properties。
* REST API（JSON）與後端溝通，HTTP-only Cookie 承載 Session Token。
* 原則上不引入第三方前端套件；必要時鎖定版本並使用 SRI Hash 驗證。

### 11.4 郵件發送

* 使用 Python 標準 `smtplib` 或輕量封裝（如 `aiosmtplib`）發送郵件，**不引入完整的第三方郵件服務 SDK**（如 SendGrid、Mailgun）以維持自主控制。
* SMTP 連線需支援 STARTTLS 或 SSL/TLS；禁止使用明文 SMTP 連線。
* 郵件範本以純文字或最小化 HTML 撰寫，禁止在郵件中嵌入外部追蹤像素或外連資源。

### 11.5 部署與環境

* **環境區隔**：開發（dev）、測試（test）、生產（prod）環境的配置需完全分離，透過環境變數注入差異設定。
* **機密管理**：資料庫路徑 / DSN、SMTP 密碼等機密值**嚴禁硬編碼於程式碼或版本控制中**，一律透過環境變數或受保護的 Secret 管理工具提供。
* **依賴鎖定**：前後端所有依賴套件需明確鎖定版本（Python 端使用 `requirements.txt` 或 `pyproject.toml` + lock 檔），確保部署環境可重現。
* **HTTPS**：生產環境前端與 API 服務一律透過 HTTPS 提供，HTTP 請求須強制重導向至 HTTPS；禁止在生產環境關閉 SSL/TLS 憑證校驗。

## 12. 關鍵設計決策 (Key Design Decisions)

本節明訂在開發啟動前必須確認的架構決策，以防止跨模組實作方向衝突。

### 12.1 任務執行觸發方式與程序模型

* **觸發方式**：任務建立後**不自動啟動**；使用者須於前台任務詳情頁面點擊「啟動」按鈕，向後端 API 發送啟動請求後，任務才進入執行狀態。
* **執行程序模型**：爬蟲核心以**獨立子程序（Subprocess）** 方式由 Web 服務於收到啟動請求時 fork / spawn，與 Web 服務進程分離執行。
  * Web 服務本身不阻塞等待爬蟲完成，任務狀態由爬蟲子程序直接寫入 Crawler DB。
  * Web 服務透過輪詢 Crawler DB 任務狀態欄位，向前台回應任務進度。
  * 每個任務同一時間只允許一個爬蟲子程序執行（藉由執行狀態安全鎖，詳見 §4）。
* **暫停 / 停止信號**：Web 服務透過作業系統信號（如 SIGTERM）或共享狀態欄位向爬蟲子程序傳遞暫停指令，爬蟲在完成當前網址爬取後安全終止。

### 12.2 初始管理員帳號 Bootstrap 機制

邀請制系統中，第一個管理員帳號須透過以下機制建立，而非透過前台或後台介面：

* **CLI 指令**：提供專用 CLI 指令 `python cli.py --create-admin <email>`，執行後在 Auth DB 中直接建立一個狀態為 `Active`、角色為 `Admin` 的帳號，並在終端機輸出系統隨機產生的高強度初始密碼。管理員可直接以此密碼進行登入，並於登入後至個人設定中自行修改為自訂安全密碼。
* **安全限制**：該 CLI 指令僅允許在伺服器本機端執行（不透過 API 暴露），且執行時需確認 Auth DB 尚未存在任何 Admin 帳號（防止重複建立）。
* **後續管理員新增**：第一位管理員建立後，後續管理員帳號統一透過後台「角色管理」功能指派，不再使用 CLI 指令。

### 12.3 Session Token 有效期與續期策略

* **有效期限**：Session Token 預設有效期為 **8 小時**（可於環境變數中配置）。
* **續期機制（Sliding Window）**：使用者在 Token 有效期內持續操作時，後端在每次 API 請求成功後自動將 Token 有效期順延（重置為 8 小時），無需使用者重新登入。
* **強制過期**：以下情況強制使 Token 立即失效，不論是否在有效期內：
  * 使用者主動登出。
  * 管理員停用（Suspend）該帳號。
  * Token 已超過最大絕對有效期（如 7 天，不論是否持續操作）。
* **同時登入**：系統允許同一帳號在多個裝置上同時持有有效 Session（`sessions` 表中可存在多筆有效記錄），各 Session 獨立管理有效期。

### 12.4 跨資料庫 `user_id` 一致性策略

由於 Auth DB 與 Crawler DB 實體分離（§5.1），兩者之間**不得執行跨庫 JOIN**，以下策略確保資料一致性：

* **`user_id` 為 Auth DB 中 `users.id` 的值**，在任務建立時由 Web 服務層從已驗證的 Session 中取得後，寫入 Crawler DB 的 `jobs.user_id` 欄位。Crawler DB 不儲存 any 其他使用者資料（如 email、角色），僅儲存此 ID 以供任務歸屬識別。
* **顯示層組合**：前台 / 後台需呈現「任務所屬使用者 email」時，由 Web 服務層分別查詢兩個資料庫後在應用層組合，禁止在資料庫層跨庫操作。
* **帳號刪除的孤立資料處理**：管理員刪除帳號時，Web 服務層須同步清除 Crawler DB 中該 `user_id` 的所有任務及結果資料（在同一 API 事務中完成，先刪 Crawler DB 資料，再刪 Auth DB 帳號），確保不產生孤立資料。

### 12.5 任務狀態變更 Email 通知機制

當長時間運行的爬蟲任務結束或發生例外中斷時，系統需提供主動通知機制，避免使用者頻繁手動輪詢：

* **觸發時機**：任務狀態轉移至 `completed`（任務正常結束，包含達到最大頁數限制或待爬佇列清空）或 `error`（任務因未預期例外異常終止）時自動觸發。
* **資料庫聯動取得 Email**：在爬蟲進程中，藉由任務的 `user_id`，利用 `backend.auth.db` 在 Auth DB 中查詢取得該使用者的 Email 信箱。若為匿名任務 (`user_id` 為 None)，則不發送通知。
* **統計資訊組裝**：通知郵件（純文字與 HTML 格式）會附帶任務的詳細資訊，並動態統計該任務於 Crawler DB 中尋獲的外部連結狀態，包含總外連數、損壞連結 (Broken Links，HTTP 狀態碼 >= 400) 的數量、與失效連結 (Dead Links，DNS 解析失敗) 的數量。
* **獨立運作防禦性**：郵件發送完全基於 `backend/email_sender.py` 的 SMTP 連線。為確保核心獨立運作原則 (CLI-First)，若爬蟲核心執行於無後端模組的純單機環境中（ImportError），發信模組應能被防禦性跳過，並於日誌中警告，而不影響爬蟲子程序的執行。

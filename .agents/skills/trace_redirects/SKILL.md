# 追蹤連結跳轉過程 (Trace Redirects)

這項技能允許你分析一個網址在收到請求後，中間經歷了多少次的跳轉，以及最後實際落地的最終網址與狀態。這對於排查「無效重導向」、「無限迴圈重導向」或是「惡意跨域跳轉」非常有幫助。

## 如何執行追蹤

當使用者要求追蹤網址跳轉歷程時，你必須依序使用以下兩種工具（標準測試與專案腳本），以便交叉比對系統反應。

**⚠️ 遵守環境規範 (Virtual Environment Rule)**：
呼叫專案腳本時，必須使用專案的虛擬環境直譯器執行。

### 步驟 1：標準跳轉行為基準 (cURL)
首先，使用 `curl` 作為基準，取得目標網址的原始、未經專案邏輯處理的跳轉路徑：
```bash
curl -c ./curl_cookies.txt -b ./curl_cookies.txt -v -L "<欲測試的網址>"
```
觀察輸出的 HTTP Header (尤其是 `Location` 與 HTTP 狀態碼)，了解伺服器的原生行為。

### 步驟 2：本專案腳本測試
接著，根據目標網址的性質，使用本專案內建的腳本再次測試。這能讓你觀察本專案的核心爬蟲元件 (`CrawlerCore`) 實際上是如何處理該網址。
- 如果是**外部連結**：
  ```bash
  .venv/bin/python scripts/test_ext.py "<欲測試的網址>"
  ```
- 如果是**內部網頁**：
  ```bash
  .venv/bin/python scripts/test_url.py "<欲測試的網址>"
  ```

*(註：如需測試特殊的延遲跳轉或追蹤 Cookie 變化，可搭配原本的 `.venv/bin/python .agents/skills/trace_redirects/scripts/trace.py "<欲測試的網址>"` 進行進階排查。)*

### 步驟 3：產出差異比較報告
當你取得上述命令輸出結果後，請將跳轉過程整理成清晰的條列式或 Markdown 區塊回報給使用者。
**最重要的是**，你必須比較「步驟 1 (curl)」與「步驟 2 (專案腳本)」的反應。
- 如果兩者結果不同（例如 curl 成功取得網頁，但專案腳本回報 403 或連線超時），必須主動分析並說明可能的原因。
- 說明本專案的程式反應與標準的 `curl` 程序有什麼不同，例如是否有 Cookie 處理、User-Agent 攔截或是憑證機制的差異。
